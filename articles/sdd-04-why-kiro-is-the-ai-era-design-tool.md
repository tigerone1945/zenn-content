# なぜCursorやClaude CodeではなくKiroなのか

前回の記事では、仕様駆動開発（SDD）について解説しました。

AI時代では、コードではなく仕様が正本になる。

これがSDDの考え方です。

すると次の疑問が出てきます。

> 仕様が重要なのは分かった。
>
> でも、実際に仕様はどう書けばいいのだろう？

今回はその話です。

---

# AI開発ツールは実装支援が中心

最近のAI開発ツールとしては、

- Cursor
- Claude Code
- Codex

などが有名です。

私自身も利用しています。

これらは非常に優秀です。

なぜなら、コードを書く作業を大幅に削減できるからです。

---

# しかし彼らが得意なのは実装である

Cursorも、Claude Codeも、基本的には実装支援ツールです。

つまり、仕様が存在することが前提です。

例えば、

「問い合わせ分類AIを作って」

という依頼はできます。

しかし、

- 分類は何種類なのか
- 優先度は何段階なのか
- 要約は何文字なのか

までは決めてくれません。

決めるのは人間です。

---

# 本当に難しいのは仕様である

実は、開発で一番難しいのは実装ではありません。

仕様です。

なぜなら、実装には正解があります。

しかし、仕様には正解がありません。

---

# Difyでも同じ問題が起きる

これはDify利用者にも共通します。

最初は簡単なワークフローだったのに、気が付くと分岐だらけになっている。

さらに、

- なぜこのプロンプトにしたのか
- なぜこのLLMノードを置いたのか

が分からなくなることがあります。

その時に困るのは、ワークフローの実装方法ではありません。

当時の判断基準です。

つまり必要なのは、コード管理ではなく仕様管理なのです。

---

# 仕様を書くためのツールが必要になる

ここで初めて、仕様作成ツールの話になります。

AI時代は、コードを書くコストより仕様を整理するコストの方が高くなります。

だから私は、実装ツールよりも先に仕様作成ツールが重要になると考えています。

---

# Kiroとは何か

Kiroは、仕様駆動開発（SDD）を支援するツールです。

CursorやClaude Codeが実装を支援するのに対して、Kiroは仕様作成を支援します。

つまり、

Kiro → 仕様作成

Claude Code → 実装

という役割分担が可能になります。

---

# なぜ今Kiroなのか

私は、Kiroが優れているから使うのではありません。

仕様を中心に考えるなら、仕様作成ツールが必要になる。

その考え方に最も近いツールが、現時点ではKiroだと考えています。

---
