## 2-1 よくあるPoC失敗パターン

### PoCは動いた。でも、そこで終わった。

AI導入の現場でよく聞く話があります。

「検証環境では精度が出た。でも本番に持っていったら全然使えなかった」
「エンジニアがシステムを作り切った。でも現場のスタッフが使ってくれない」
「最初の2週間は使われた。でも1ヶ月後には誰も触っていなかった」

PoC（Proof of Concept：概念実証）は「できる」を示すための小さな実験です。問題は、PoCが成功しても、**その先の「定着」に失敗するケースが非常に多い**ことです。

失敗には、大きく3つのパターンがあります。

---

### 失敗パターン① 精度がズレる

「精度が出ない」という相談の中で最も多いのが、**「検証時と本番時で結果が変わる」** というケースです。

よくある原因はこうです。

**テストデータと本番データが違いすぎる**

検証時に使ったデータは、整理されたサンプルデータや過去のきれいなデータ。でも本番のデータは、誤字脱字あり、口語あり、文脈が省略されていたり、想定外の書き方をしている。

```
検証時のデータ：
「商品が届いていません。注文番号は12345です。至急対応をお願いします。」

本番のデータ：
「まだこない。12345。はやくして」
「届いてないんですけど昨日注文したやつ、配送の問題ですか？それとも在庫？」
「商品の件なんですが…少し複雑な話で、まず状況を説明すると…（長文が続く）」
```

AIは、学習や検証に使ったデータの傾向に引っ張られます。本番データが検証データから遠ければ遠いほど、精度は落ちます。

**プロンプトが本番の多様性に対応できていない**

「こういう入力が来たとき、こう出力してください」というプロンプトは、検証時に想定したパターンにしか対応できません。実務では、想定外の入力が必ず来ます。

対策として重要なのは、**本番に近いデータでテストする** こと。そして **出力を固定する**（後述するJSON出力）ことです。

---

### 失敗パターン② 現場で使われない

「システムは完成した。でも誰も使わない」——これはAI導入あるある中のあるあるです。

**使われない理由は、たいてい技術的な問題ではありません。**

よくある原因を正直に列挙します。

- **現場の人が作成に関わっていなかった**
  → 「何が課題か」を現場に聞かず、開発者の想像で作った
  → できたものが現場の実際の業務フローと合っていない

- **操作が既存の業務フローに合っていない**
  → 「AIを使うために、別のツールを開いて、コピペして…」という手間が発生
  → 今まで通りやったほうが早い、と判断される

- **信頼されていない**
  → AIが出した結果を確認する手段がない
  → 「本当に正しいの？」という不安から、結局手動で確認している

- **責任の所在が曖昧**
  → AIが判断を間違えたとき、誰が責任をとるのかが決まっていない
  → 「AIのせいにできない」から、使わないほうが安全と判断される

現場で使われるAIを作るためには、**現場の人を最初から巻き込む**ことが必須です。技術的な完成度より、「現場が使いたいと思えるか」のほうが重要です。

---

### 失敗パターン③ 運用されない

「作ったけど、メンテできなくて止まった」

これも非常に多い失敗です。

AI活用の現場では、こんなことが起きます。

- AIの出力精度が下がってきた。でも、プロンプトを誰が直すかわからない
- Difyのバージョンが上がって動作が変わった。でも対応できる人がいない
- 使っているモデルが廃止・変更された。誰も気づかなかった
- 担当エンジニアが退職して、コードの意図が誰にもわからない

運用を考えずに「動くものを作る」だけに集中すると、**作った瞬間が最も完成度が高く、あとは劣化していく一方**になります。

運用設計のポイントは4つです。

```
① 担当者を決める
   → 誰がプロンプトをメンテするのか
   → 誰が精度を監視するのか

② ログを残す
   → AIへの入力と出力をSpreadsheetに記録する
   → 「いつ何が起きたか」を後から確認できるようにする

③ 異常を検知する
   → AIの出力がおかしいとき、誰かが気づける仕組みを作る

④ 非エンジニアが触れる設計にする
   → Difyを使う理由のひとつはここ
   → プロンプトの変更くらいは、現場のリーダーがGUIで対応できる状態にする
```

本Bookの演習では、**Spreadsheetにログを残す** という設計を標準で入れています。これは「うまく動いているか」を見えるようにするための最低限の仕組みです。

## 2-2 RAGが失敗する理由

### RAGへの過剰な期待

最近、AI導入のプロジェクトで「RAGを使えば解決する」という話をよく聞きます。

RAG（Retrieval-Augmented Generation）とは、外部の知識データベースを参照しながら回答を生成する仕組みです。「社内ドキュメントをAIに読ませて、質問に答えさせる」というユースケースでよく使われます。

RAGは強力な技術です。ただ、**正しく設計しないと、ほぼ確実に失敗します。**

---

### RAG失敗理由① データ構造問題

RAGは「適切なドキュメントを検索して、その内容を元に回答する」仕組みです。

つまり、**データが整理されていないと、そもそも正しいドキュメントが検索できません。**

よくある状況：

```
社内に存在するドキュメント：
- 3年前に作られた規程集（Wordファイル、更新履歴なし）
- 複数バージョンが混在するマニュアル（どれが最新か不明）
- 担当者しか知らないローカルルール（どこにも文書化されていない）
- 会議の議事録（内容が散漫で、何が決定事項か不明）
```

このまま全部RAGに投入しても、AIは「どのドキュメントを信頼すればいいか」わかりません。古い情報と新しい情報が混在していれば、古い情報を元に回答することもあります。

**RAGを使う前に、まずデータの整理が必要。**
でも多くのプロジェクトは、データ整理をスキップしてRAGを構築しようとします。これが最初の躓きです。

---

### RAG失敗理由② PPTそのまま投入問題

「社内の資料をそのままRAGに食わせればいいんでしょ？」

これは大きな誤解です。

PowerPointやPDFをそのままRAGに投入すると、こんな問題が起きます。

**スライド1枚のテキスト例：**
```
【Q3戦略について】
・売上目標：前年比120%
・重点施策3点
  ①新規顧客開拓
  ②既存顧客深耕  
  ③コスト最適化
```

このスライドには「なぜ120%なのか」「具体的にどう開拓するのか」という文脈がありません。スライドは「発表者の口頭説明」と一緒に使われることを前提に作られているからです。

RAGはテキストを読みますが、**スライドに書いていない文脈（口頭説明・前後のスライドとの関係・作成時の背景）** は読めません。結果として、表面的な断片情報しか持てず、質問に対して的外れな回答をします。

RAGに投入するドキュメントは、**「テキストだけで文脈が完結しているもの」** である必要があります。スライドやビジュアル資料は、テキストに変換・補足してから投入する必要があります。

---

### RAG失敗理由③ Chunk分割問題

RAGは、ドキュメントを「Chunk（チャンク）」と呼ばれる小さなブロックに分割して管理します。質問が来たとき、関連するChunkを検索して回答の元データにします。

このChunkの分割方法が、精度に直結します。

**悪い分割例：**
```
元のテキスト：
「返品ポリシーについて。商品到着後7日以内であれば返品を承ります。
ただし、食品・消耗品・カスタマイズ商品は対象外です。
返品の際は、必ず事前にカスタマーサポートへご連絡ください。
返送料はお客様負担となります。」

300文字でChunk分割した場合：
[Chunk1]「返品ポリシーについて。商品到着後7日以内であれば返品を承ります。ただし、食品・消耗品・カスタマイズ商品は対象外です。返品の際は、必ず事前にカスタマーサポートへご連」
[Chunk2]「絡ください。返送料はお客様負担となります。」
```

Chunk2だけが検索にヒットした場合、「返送料はお客様負担」という情報は取得できますが、「7日以内」という条件や「カスタマイズ商品は対象外」という重要情報が欠落します。

**Chunk分割は「文字数」ではなく「意味のまとまり」で行う**のが正しいアプローチです。ただしこれは設計コストが高く、ドキュメント構造への深い理解が必要です。

---

### RAGのまとめ

RAGは「魔法の箱」ではありません。使いこなすには、相応のデータ設計が必要です。

```
RAGがうまくいく条件：
✅ ドキュメントが整理・最新化されている
✅ テキストだけで文脈が完結している
✅ Chunk分割が意味単位で適切に設計されている
✅ 定期的にデータをメンテできる体制がある

RAGがうまくいかない条件：
❌ 古いドキュメントと新しいドキュメントが混在している
❌ スライドや図表が多い資料をそのまま投入している
❌ Chunkが文字数ベースで機械的に分割されている
❌ 一度作ったらメンテしない
```

本Bookの演習では、RAGは使いません。代わりに **「Spreadsheetにデータを整理して、GASで取得してAIに渡す」** というシンプルな設計を採用しています。このほうが、初学者にとっても実務でも、圧倒的に安定します。

## 2-3 「それっぽいAI」の危険性

### 「なんとなく動いている」AIの怖さ

AIが出す回答を見たとき、こんな感想を持ったことはありませんか？

「なんか、それっぽいことを言ってるけど、本当に正しいのかな？」

この感覚は正しい疑いです。AIには「それっぽく見えるが、実は間違っている」という出力を平然とする特性があります。これを放置すると、実務での信頼を失います。

---

### 危険① ハルシネーション

ハルシネーションとは、AIが「事実ではないことを自信満々に答える」現象です。

```
ユーザー：「○○法という法律の第3条には何と書いてありますか？」

AI：「○○法第3条には『事業者は、消費者に対して〜〜の義務を負う』と規定されています。」

→ 実際には、そのような条文は存在しない。
  AIが「それっぽい条文」を生成してしまった。
```

ハルシネーションは、知識が足りないときに起きやすいですが、知識があるときでも起きます。特に **「細かい数字・固有名詞・条文・手順」** が絡む場合は要注意です。

実務でのリスクを下げるためには：

- **AIに「事実の確認」をさせない**。AIは「生成」は得意ですが「検索・照合」は苦手。
- **出力を固定する**。「はい/いいえ」「A/B/C/D」のように選択肢を与えると、ハルシネーションは起きにくくなる。
- **ログを残して、人間がチェックできる仕組みを作る**。

---

### 危険② 曖昧出力

AIに「このメールを分類して」とだけ頼むと、こんな出力が返ってくることがあります。

```
このメールは、配送に関する問い合わせとクレームの要素が混在しており、
どちらかに分類することが難しい内容です。
配送の遅延に対する不満（クレーム要素）と、
現在の配送状況を知りたいという問い合わせ（問い合わせ要素）の
両方が含まれています。
文脈や担当者の判断によって対応が異なるかもしれません。
```

文章としては丁寧で正確です。でも、**システムとしては使い物になりません。**

プログラムで扱いたいのは「クレーム」「問い合わせ」のような固定値です。上記のような自然言語の回答は、if文での分岐もデータベースへの保存もできません。

これが「曖昧出力」の問題です。

解決策は第4章で詳しく説明しますが、ひとことでいうと **「JSON形式で出力を固定する」** です。

```json
{
  "category": "クレーム",
  "sub_category": "配送遅延",
  "confidence": 0.8
}
```

このように出力を固定することで、プログラムで扱えるデータになります。

---

### 危険③ 出力揺れ

同じ入力を与えても、AIの出力は毎回少しずつ変わります。

```
入力：「このレビューの感情を分析して：『商品は良いけど、
        梱包が雑でした』」

1回目の出力：「ポジティブ（商品評価）とネガティブ（梱包）が混在」
2回目の出力：「ネガティブ（全体的に不満が残る内容）」
3回目の出力：「中立（良い面と悪い面の両方を指摘している）」
```

この「出力揺れ」があると、システムとして信頼できません。同じデータを処理しても、今日と明日で結果が変わるようなシステムは実務では使えません。

出力揺れを抑える主なアプローチ：

1. **Temperatureを下げる**：AIの「創造性・ランダム性」パラメータを低く設定する
2. **JSON出力を強制する**：選択肢を固定して自由度を下げる
3. **Few-shotプロンプトを使う**：「こういう入力にはこう答えよ」という例を示す

本Bookの演習では、これらをすべて組み合わせて使います。

## 2-4 AI導入で本当に重要なこと

ここまでの失敗パターンを整理すると、AI導入でつまずく理由は大きく2つに集約されます。

1. **技術的な問題**：精度が出ない、出力が不安定、RAGの設計ミス
2. **組織的な問題**：現場に定着しない、運用が続かない、責任が曖昧

技術的な問題は、設計で解決できます。でも、多くのプロジェクトはそれより前に「何を作るか」「誰が使うか」「どう運用するか」という設計を怠って失敗しています。

最後に、AI導入で本当に重要な3つの原則を紹介します。

---

### 原則① 小さく始める

「全部を一気に自動化しよう」は失敗の入口です。

最初にやるべきことは、**業務の中の「一番シンプルで、効果がわかりやすいタスク」を1つ選んで、それだけを自動化する**ことです。

```
❌ 悪い始め方：
「お問い合わせの受付から回答まで全部自動化したい」
→ 範囲が広すぎる。どこで失敗したかわからない。

✅ 良い始め方：
「お問い合わせを受信したら、まず『カテゴリ分類』だけをAIにやらせる」
→ 小さい。成功失敗がはっきりわかる。改善しやすい。
```

小さく始めることで、**現場に「これ、ちょっと楽になった」という成功体験**を作れます。この成功体験が、次のステップへの信頼と推進力になります。

---

### 原則② 運用を考えてから作る

「誰がメンテするのか」が決まる前に作り始めないでください。

作ってから運用者を探そうとすると、「作った本人しか触れないシステム」ができあがります。そしてその人が異動・退職すると、誰もメンテできなくなります。

作る前に決めておくこと：

```
- プロンプトを変更するのは誰か？
- AIの精度を定期的に確認するのは誰か？
- 異常を検知したとき、最初に連絡を受けるのは誰か？
- ログはどこに残すか？誰が見るか？
```

本Bookでは、これらを「Spreadsheet中心設計」で解決します。ログをSpreadsheetに残し、プロンプトはDifyのGUIで管理する。非エンジニアでもメンテできる構成です。

---

### 原則③ 完璧を目指さない

AIに100%の精度を求めると、永遠にリリースできません。

実務のAIシステムに必要なのは **「人間が確認・修正できる仕組みの中で、十分に役立つ精度」** です。

```
目指すべき姿：
✅ AIが80〜90%の作業を自動化する
✅ 残りの10〜20%は人間がレビュー・修正する
✅ 人間の作業量が大幅に減る
✅ AIが間違えても、すぐ気づいて修正できる

避けるべき姿：
❌ AIが100%完璧に処理する（そのためのコストが膨大）
❌ 人間のチェックがない（間違いに気づけない）
❌ ひとつの間違いで全体への信頼が失われる
```

「AIは間違える」を前提に設計することが、実務AIの鉄則です。

間違えたときに気づける仕組み（ログ・通知・人間のレビュー）を作っておけば、完璧じゃなくても十分に価値のあるシステムになります。

---

> **第2章のまとめ**
>
> - PoCの失敗は「精度がズレる」「現場で使われない」「運用されない」の3パターン
> - RAGはデータ整理・Chunk設計・運用体制が整って初めて機能する
> - ハルシネーション・曖昧出力・出力揺れはJSON固定出力で大幅に軽減できる
> - AI導入の原則は「小さく始める」「運用を考えてから作る」「完璧を目指さない」
> - 「AIは間違える」を前提に設計することが実務AIの鉄則
