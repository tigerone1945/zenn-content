## 3-1 GASが強い理由

### GASとは何か

GAS（Google Apps Script）は、Googleが提供するJavaScriptベースのスクリプト環境です。

Googleアカウントがあれば誰でも無料で使えます。特別な開発環境のセットアップも不要で、ブラウザ上でコードを書いてそのまま実行できます。

「JavaScriptが書ける人向けでしょ？」と思うかもしれませんが、本Bookの演習レベルであれば、プログラミング初心者でも習得できます。コードの意味がわからなくても、動作する理由を理解しながら進めることができます。

---

### GASの強み① Google Workspaceとの相性

GASが最も輝くのは、**Google Workspaceとの連携**です。

```javascript
// Spreadsheetのデータを取得する（たった3行）
const sheet = SpreadsheetApp.getActiveSpreadsheet().getActiveSheet();
const data = sheet.getDataRange().getValues();
```

たったこれだけで、Spreadsheetのデータを取得できます。

Python + pandas + Google Sheets APIで同じことをやろうとすると、認証の設定だけで30分以上かかります。GASなら、Spreadsheetを開いてスクリプトを書けばすぐ動く。

同様に、Gmail・Google フォーム・Google Drive・Google カレンダーとも、数行で連携できます。

```javascript
// Gmailの未読メールを取得する
const threads = GmailApp.search('is:unread');

// Googleフォームの回答を取得する
const form = FormApp.openById('フォームID');
const responses = form.getResponses();

// Google Driveにファイルを保存する
const folder = DriveApp.getFolderById('フォルダID');
folder.createFile('report.txt', content);
```

**Google Workspaceを使っている会社なら、GASは最も低コストで最もスムーズに動く自動化ツールです。**

---

### GASの強み② Spreadsheet中心設計との相性

本Bookでは「Spreadsheet中心設計」を採用しています。

これは、**データ管理・ログ管理・設定管理をすべてSpreadsheetで行う**という設計思想です。

なぜSpreadsheetを中心に置くのか？

- **誰でも見られる**：エンジニアでなくても、Spreadsheetなら開いて確認できる
- **ログが残る**：AIの入出力履歴をそのまま記録できる
- **設定を変更しやすい**：閾値や分類ルールをセルに書けば、コードを変えずに動作を変えられる
- **共有しやすい**：チーム全員がリアルタイムで同じデータを参照できる

```
Spreadsheetをハブにした設計の例：

[入力データシート]：処理対象のデータ（レビュー・応募情報など）
[出力結果シート]　：AIの処理結果・スコア・判定
[ログシート]　　　：実行日時・入力・出力・エラー
[設定シート]　　　：APIエンドポイント・閾値・分類ラベル
```

この設計にすることで、**非エンジニアが運用できる**システムになります。プロンプトを変えたいときはDifyのGUIで、閾値を変えたいときはSpreadsheetのセルを変えるだけ。コードに触らずに多くの変更ができます。

---

### GASの強み③ 非エンジニアでも扱いやすい

GASのコードはシンプルで読みやすいです。

```javascript
function analyzeReviews() {
  // 1. Spreadsheetからレビューデータを取得
  const sheet = SpreadsheetApp.getActiveSpreadsheet().getSheetByName('レビュー');
  const reviews = sheet.getDataRange().getValues();
  
  // 2. 各レビューをDify APIに送信して分析
  for (let i = 1; i < reviews.length; i++) {
    const reviewText = reviews[i][0];
    const result = callDifyAPI(reviewText);
    
    // 3. 結果をSpreadsheetに書き込む
    sheet.getRange(i + 1, 2).setValue(result.sentiment);
    sheet.getRange(i + 1, 3).setValue(result.score);
  }
}
```

コメントさえ書いておけば、「何をしているか」が非エンジニアでも読み取れます。

また、GASには**実行ログ**が標準で備わっています。エラーが起きたとき、何が問題だったかをすぐ確認できます。

---

### GASの注意点

正直に弱点もお伝えします。

**実行時間の制限（6分）**
GASは1回の実行で6分以上かかる処理はタイムアウトします。大量のデータを一括処理する場合は、分割実行の設計が必要です。

**リアルタイム処理は苦手**
「メールが届いた瞬間に処理する」という厳密なリアルタイムは難しい。トリガーの最短実行間隔は1分です。

**複雑な処理には限界がある**
機械学習モデルの学習・大規模データ処理・複雑なビジネスロジックは、Pythonや本格的なバックエンドのほうが適しています。

本Bookの演習は、これらの制限の範囲内で動作するように設計しています。

## 3-2 Difyが強い理由

### Difyとは何か

Difyは、LLM（大規模言語モデル）を使ったアプリケーションを構築するためのオープンソースプラットフォームです。

ひとことで言えば、**「AIアプリをGUIで設計・管理できるツール」** です。

プロンプトを書いてコードに埋め込む、ということをせずに、Difyのダッシュボード上でAIの動作を設計・管理・更新できます。

無料枠で始められ、クラウド版（dify.ai）ならアカウントを作ればすぐ使えます。

---

### Difyの強み① ワークフロー設計

Difyの「ワークフロー」機能は、AIの処理ステップをビジュアルで繋いで設計できます。

```
ワークフローの例（採用スクリーニング）：

[開始]
  ↓
[変数入力]：応募者情報 / 求人要件
  ↓
[LLMノード1]：スキル抽出（JSON出力）
  ↓
[LLMノード2]：スコアリング（JSON出力）
  ↓
[条件分岐]：スコア70以上 → 「推奨」/ 50〜69 → 「要検討」/ 50未満 → 「不合格」
  ↓
[終了]：最終判定JSONを返す
```

このフローをコードで書かずにGUIで組める。これがDifyのワークフロー機能です。

処理が複雑になっても、ビジュアルで全体像を把握できるため、**作った本人以外でも構造を理解しやすい**という利点があります。

---

### Difyの強み② プロンプト管理

コードにプロンプトを直書きすると、こんな問題が起きます。

```javascript
// ❌ コードに直書きしたプロンプト
const prompt = `あなたはレビュー分析の専門家です。
以下のレビューを分析して、感情を分類してください。
...（長いプロンプト）...`;
```

- プロンプトを変えるたびにコードのデプロイが必要
- プロンプトの変更履歴が管理できない
- 非エンジニアがプロンプトを変更できない

Difyでは、プロンプトはGUIで管理されます。

- GUIからプロンプトを変更できる
- バージョン管理が標準でついている
- 変更のロールバックが簡単
- 非エンジニアでもプロンプトを調整できる

「AI担当のエンジニアがいないと何もできない」状態から脱出できます。

---

### Difyの強み③ JSON固定出力

Difyでは、AIの出力形式をJSONに固定できます。

```json
{
  "sentiment": "ネガティブ",
  "score": 2,
  "issues": ["梱包が雑", "配送が遅い"],
  "requires_action": true,
  "priority": "高"
}
```

この機能が、実務AIで特に重要です。

出力が自然言語の文章だと、GASでパース（解析）しにくく、Spreadsheetへの書き込みも難しい。JSONで固定されれば、どのキーに何の値が入るかが確定しているため、プログラムで扱いやすくなります。

また、出力形式を固定することで **「出力揺れ」** も大幅に軽減されます。

---

### Difyの注意点

Difyにも弱点があります。

**カスタマイズの限界**
複雑な処理（カスタムロジック・既存システムとの深い統合）は、Difyのワークフローだけでは対応しきれないことがあります。

**クラウド版の無料枠の制限**
無料プランにはメッセージ数の制限があります。本番運用では有料プランの検討が必要です。

**学習コスト**
初めて触るときは、ノード・ワークフロー・変数の概念に慣れる時間が必要です。

ただし、これらは本Bookの演習の範囲では問題になりません。PoCから始めて、本番運用が見えてきた段階でスケールを考えれば十分です。

## 3-3 この構成が実務向きな理由

### 「作れる × 使える × 続けられる」の三拍子

AIシステムを実務に定着させるには、3つの条件がそろう必要があります。

| 条件 | 意味 | GAS × Difyでの解決 |
|---|---|---|
| **作れる** | 技術的に実現できること | GASは低コスト・Difyは直感的なGUI |
| **使える** | 現場のフローに組み込めること | SpreadsheetやSlackと直接連携できる |
| **続けられる** | 非エンジニアでもメンテできること | プロンプトはDify、設定はSpreadsheetで管理 |

PythonやLangChainは「作れる」という点では強力ですが、「使える」「続けられる」を満たすためのインフラ設計・運用体制が別途必要になります。

GAS × Difyは、この3条件を**小規模チームでも達成しやすい**という点で実務向きです。

---

### 小規模導入しやすい

GAS × Difyは、初期コストがほぼゼロです。

```
コスト試算（月次・最小構成の場合）：

GAS           : 無料（Google Workspaceに含まれる）
Dify（クラウド）: 無料枠から開始可能
OpenAI API    : 使った分だけ（小規模なら月数百円〜数千円程度）
Slack         : 無料プランで通知機能は使える

合計：最小構成なら月数百円〜数千円で始められる
```

「まずPoCをやってみて、効果が出たら投資を増やす」というステップが踏みやすい構成です。

---

### PoCしやすい

GAS × Difyでは、アイデアから動くものまでのサイクルが短いです。

```
典型的な開発サイクル：

Day1：Difyでプロンプトを設計してAPIを叩けるようにする
Day2：GASでSpreadsheetからデータを取得してDifyに送るコードを書く
Day3：出力をSpreadsheetに書き戻す処理を追加する
Day4：Slack通知を追加する
Day5：テスト・調整・動作確認

→ 1週間で動くPoCが完成する
```

大規模なシステム開発では、環境構築・設計・レビュー・テストに数週間かかることも珍しくありません。GAS × Difyなら、まず動かして、フィードバックを元に改善するサイクルを素早く回せます。

## 3-4 全体アーキテクチャ

### 本Bookで使うシステム全体図

演習を通じて使うコンポーネントと、それぞれの役割を整理します。

```
┌─────────────────────────────────────────┐
│              データ入力層                  │
│  Googleフォーム / Spreadsheet / Gmail    │
└───────────────┬─────────────────────────┘
                │ データ取得
┌───────────────▼─────────────────────────┐
│              処理制御層（GAS）              │
│  ・データ取得・整形                         │
│  ・API呼び出し（Dify Webhook）              │
│  ・レスポンス解析（JSON parse）             │
│  ・Spreadsheetへの書き込み                 │
│  ・Slack通知                               │
│  ・時間トリガー管理                         │
└───────────────┬─────────────────────────┘
                │ HTTP POST（JSON）
┌───────────────▼─────────────────────────┐
│              AI処理層（Dify）              │
│  ・System Promptの管理                    │
│  ・ワークフロー設計                         │
│  ・LLM（GPT-4oなど）の呼び出し             │
│  ・JSON固定出力                            │
└───────────────┬─────────────────────────┘
                │ JSON Response
┌───────────────▼─────────────────────────┐
│              出力・通知層                  │
│  Spreadsheet（結果保存）/ Slack（通知）     │
└─────────────────────────────────────────┘
```

### 各コンポーネントの役割

**① データ入力層**

処理対象のデータが入ってくる場所です。

- **Googleフォーム**：応募情報・問い合わせなど、外部からの入力
- **Spreadsheet**：過去のデータ・マスターデータ・設定値
- **Gmail**：メールで届いた問い合わせや通知

**② GAS（処理制御層）**

全体のオーケストレーター（指揮者）です。

GASは「AIに何をいつ渡すか」「結果をどこに書くか」「誰に通知するか」を制御します。AIの「頭脳」ではなく、「手足と段取り」を担当します。

```javascript
// GASの仕事：データを取って、AIに渡して、結果を整理して、書き込んで、通知する
function processReviews() {
  const reviews = getReviewsFromSheet();     // データ取得
  const results = callDifyAPI(reviews);      // AI呼び出し
  writeResultsToSheet(results);             // 結果書き込み
  sendSlackNotification(results);           // 通知
}
```

**③ Dify（AI処理層）**

頭脳の部分です。

GASから受け取ったデータを分析・分類・生成して、JSON形式で結果を返します。プロンプトの設計・モデルの選択・ワークフローの管理はすべてDifyのGUIで行います。

**④ Spreadsheet（出力・ログ層）**

結果を保存する場所です。

AIの処理結果・実行ログ・エラー情報をSpreadsheetに記録します。「いつ何が起きたか」を後から追跡できるようにするためです。

**⑤ Slack（通知層）**

人間に「知らせる」ための場所です。

処理の完了・要注意フラグ・エラー発生などをSlackに通知します。AIがバックグラウンドで動いていても、人間がタイムリーに把握できるようにします。

---

### データの流れを具体的に見る

演習1「商品レビュー感情分析」を例に、データがどう流れるかを確認しましょう。

```
① Spreadsheetの「レビューシート」にレビューテキストが入力される

② GASが起動（手動実行 or 時間トリガー）

③ GASがSpreadsheetから未処理のレビューを取得

④ GASがDify APIにPOSTリクエストを送信
   body: { "review": "商品は良かったですが梱包が残念でした" }

⑤ DifyのワークフローがLLMに処理を依頼

⑥ LLMが分析して結果をJSON形式で返す
   {
     "sentiment": "混合",
     "positive_points": ["商品品質"],
     "negative_points": ["梱包"],
     "score": 3,
     "requires_action": false
   }

⑦ GASがJSONをパース（解析）

⑧ GASが各値をSpreadsheetの対応セルに書き込む

⑨ requires_actionがtrueの場合、SlackにアラートDM
```

この流れが理解できれば、本Bookの演習5つはすべて同じパターンの応用です。「何を入れて、何を出すか」「誰に知らせるか」が変わるだけで、基本構造は共通しています。
