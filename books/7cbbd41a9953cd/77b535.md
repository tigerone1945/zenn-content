## 4-1 AI設計で最重要な考え方

### 「何をAIにやらせるか」を先に決める

AIエージェントの設計で最もよくある失敗のひとつが、**「とりあえずAIに全部渡してみる」** です。

これは動きません。正確には「動いているように見えるが、使い物にならない」状態になります。

AIに何かをやらせる前に、必ず次の3つを明確にしてください。

```
① AIに渡す「入力」は何か？
   → テキスト？数値？複数の情報を組み合わせたもの？
   → どこから取得するのか？（Spreadsheet / フォーム / メール）

② AIに「何をやらせるか」？
   → 分類？要約？生成？判断？
   → 複数のタスクを一度にやらせようとしていないか？

③ AIから受け取る「出力」は何か？
   → どんな形式で返ってくるのか？
   → プログラムで扱える形式になっているか？
```

この3つが明確になっていないと、プロンプトを何度書き直しても精度は上がりません。

---

### タスクは「1AIノード1タスク」に分割する

「このメールを読んで、カテゴリを分類して、優先度をつけて、返信文を生成して」

この指示、AIには重すぎます。

複数のタスクを1つのプロンプトに詰め込むと、次のような問題が起きます。

- どのタスクかで精度がバラつく
- 出力形式が複雑になり、パースが難しくなる
- どのタスクが失敗したかの特定が難しい
- 改善しづらい

**1つのAIノードには、1つのタスクだけを担当させる。**

これが実務AI設計の基本原則です。

```
❌ 悪い設計（1ノードに複数タスク）：
   [AIノード]「分類 + 優先度判定 + 返信文生成」→ 出力が複雑・改善しにくい

✅ 良い設計（タスクを分割）：
   [AIノード1]「分類」→ category: "クレーム"
   [AIノード2]「優先度判定」→ priority: "高"
   [AIノード3]「返信文生成」→ reply_text: "〜〜〜"
```

Difyのワークフロー機能は、まさにこの「タスク分割」を視覚的に管理するために作られています。

---

### AIに「曖昧判断」をさせすぎない

AIは「曖昧さ」を許容するのが得意ですが、**実務システムで曖昧さを許容すると運用できなくなります。**

よくある例：

```
❌ 曖昧な指示：
「このお問い合わせが重要そうだったら担当者に通知してください」

→「重要そう」の基準がAIによって揺れる
→ 今日と明日で同じメールに対して違う判断をするかもしれない
→ 担当者への通知基準が一定にならない
```

```
✅ 明確な指示：
「以下の条件をすべて満たす場合、requires_notificationをtrueにしてください：
  - カテゴリが『クレーム』または『緊急問い合わせ』
  - かつ感情スコアが2以下（5段階）
  - かつ本文に『至急』『すぐに』『返金』のいずれかが含まれる」

→ 条件が明確
→ 毎回同じ基準で判断される
→ 基準を変えたいときはプロンプトを変えるだけ
```

「AIに判断させる」と「AIに条件を評価させる」は違います。後者のほうが、はるかに安定します。

## 4-2 JSON固定出力

### なぜJSONなのか

第2章でも少し触れましたが、JSON固定出力は実務AIの設計で最も重要なテクニックのひとつです。

AIが自然言語（普通の文章）で回答すると、こうなります。

```
入力：「このレビューの感情を分析して：『早く届きましたが、品質がイマイチでした』」

自然言語出力の例：
「このレビューは、配送速度に対してはポジティブな評価が見られますが、
 商品品質に対してはネガティブな感情が含まれています。
 全体的には、混合した感情を持つレビューと判断できます。」
```

この出力をプログラムで処理しようとすると：

```javascript
// ❌ 自然言語出力をパースしようとすると...
const text = response; // 長い文章が返ってくる
// 「ポジティブ」という文字列を探す？「ネガティブ」を探す？
// 「混合」という判定をif文でどう使う？
// → パースが非常に難しく、不安定
```

JSON出力なら：

```json
{
  "sentiment": "混合",
  "positive_aspects": ["配送速度"],
  "negative_aspects": ["商品品質"],
  "score": 3,
  "requires_action": false
}
```

```javascript
// ✅ JSONならパースが簡単
const result = JSON.parse(response);
const sentiment = result.sentiment;      // "混合"
const score = result.score;             // 3
const needsAction = result.requires_action;  // false

// Spreadsheetに書き込む
sheet.getRange(row, 2).setValue(sentiment);
sheet.getRange(row, 3).setValue(score);
```

**JSONはプログラムのための言語**です。AIの出力をプログラムで使うなら、JSON以外の選択肢はほぼありません。

---

### 出力形式を固定する意味

JSON形式を指定するだけでなく、**出力のスキーマ（構造）も固定する**ことが重要です。

スキーマを固定するとは、「このキーには必ずこの型の値が入る」ということを先に決めることです。

**スキーマ設計の例（感情分析）：**

```json
{
  "sentiment": "ポジティブ | ネガティブ | 混合 | 中立",
  "score": "1〜5の整数（1が最もネガティブ、5が最もポジティブ）",
  "positive_aspects": "ポジティブに言及された点のリスト（配列）",
  "negative_aspects": "ネガティブに言及された点のリスト（配列）",
  "requires_action": "true（対応が必要）| false（対応不要）",
  "priority": "高 | 中 | 低"
}
```

このスキーマを事前に決めておくと：

- プロンプトに「この形式で出力してください」と明示できる
- GASのコードで「どのキーを参照すればいいか」が確定する
- Spreadsheetの列設計もスキーマに合わせて作れる
- 出力が安定する（揺れにくくなる）

スキーマ設計は、AIの実装より先にやるべき設計作業です。

---

### Difyでの設定方法

DifyではJSON出力を強制するための設定ができます。

**System Promptへの記述例：**

```
あなたはECレビュー分析の専門家です。
与えられたレビューテキストを分析し、必ず以下のJSON形式のみで回答してください。
説明文・前置き・余分なテキストは一切含めないでください。

出力形式：
{
  "sentiment": "ポジティブ" または "ネガティブ" または "混合" または "中立",
  "score": 1から5の整数,
  "positive_aspects": ["ポジティブな点1", "ポジティブな点2"],
  "negative_aspects": ["ネガティブな点1", "ネガティブな点2"],
  "requires_action": true または false,
  "priority": "高" または "中" または "低"
}

注意事項：
- sentimentは必ず上記4つのいずれかを選択する
- scoreは必ず1〜5の整数
- positive_aspects/negative_aspectsは該当なしの場合は空配列 []
- requires_actionはネガティブ かつ score2以下の場合のみtrue
```

このように、**「何を出力するか」だけでなく「何を出力しないか」** も明示することが重要です。

## 4-3 Few-shot設計

### Few-shotとは

Few-shot（フューショット）とは、「こういう入力にはこう出力せよ」という例をプロンプトに含める手法です。

AIはゼロから推論するより、**「例を見てパターンを学ぶ」** ほうが安定した出力を出せます。

```
Zero-shot（例なし）：
「このレビューの感情を分析してJSONで返してください。
 レビュー：『思ったより小さかったですが、品質はよかったです』」

→ AIが「思ったより小さい」をどう判断するか、実行するまでわからない

Few-shot（例あり）：
「このレビューの感情を分析してJSONで返してください。

例1：
入力：「とても良い商品でした！また買います」
出力：{"sentiment": "ポジティブ", "score": 5, "requires_action": false}

例2：
入力：「全然使えない。返金してほしい」
出力：{"sentiment": "ネガティブ", "score": 1, "requires_action": true}

例3：
入力：「デザインはいいけど耐久性が低い」
出力：{"sentiment": "混合", "score": 3, "requires_action": false}

では以下を分析してください：
入力：『思ったより小さかったですが、品質はよかったです』」

→ 例を見ているので、「混合」と判定しscoreは中程度になる可能性が高い
```

---

### 良い例の作り方

Few-shotの例は、「典型例」と「境界線上の例」を組み合わせて作ります。

```
典型例（明らかなケース）：
→ ポジティブが明らかな例
→ ネガティブが明らかな例

境界線上の例（難しいケース）：
→ ポジティブとネガティブが混在している例
→ 感情が読み取りにくい例
→ 実務でよく迷うケース
```

**良いFew-shotの例：**

```
例1（典型的ポジティブ）：
入力：「配送も早くて品質も最高でした！大満足です」
出力：{"sentiment": "ポジティブ", "score": 5, "requires_action": false}

例2（典型的ネガティブ）：
入力：「届いた時点で壊れていた。最悪。絶対返品する」
出力：{"sentiment": "ネガティブ", "score": 1, "requires_action": true}

例3（混合・requires_action=false）：
入力：「値段の割にはいいけど、説明書がわかりにくい」
出力：{"sentiment": "混合", "score": 3, "requires_action": false}

例4（混合・requires_action=true）：
入力：「デザインは好きだけど、2週間で壊れた。交換希望」
出力：{"sentiment": "混合", "score": 2, "requires_action": true}
```

例3と例4の違いに注目してください。どちらも「混合」ですが、「交換希望」という対応要求があるかどうかでrequires_actionが変わります。こういった**境界線上のケースを例示すること**が、Few-shotの精度向上のカギです。

---

### 悪い例の使い方

「こういう出力をしないでください」という**悪い例（Negative Example）** も有効です。

特に、過去にAIが間違えたケースを悪い例として追加することで、同じ間違いを繰り返しにくくなります。

```
注意：以下のような出力は不正解です。

悪い例1（説明文が混入）：
「このレビューは全体的にポジティブです。なぜなら...」
→ JSON以外のテキストは絶対に含めないでください。

悪い例2（スキーマ違反）：
{"emotion": "良い", "rating": "高"}
→ sentimentとscoreのキー名と型を必ず守ってください。

悪い例3（範囲外の値）：
{"sentiment": "とてもネガティブ", "score": 0}
→ sentimentは4択のみ、scoreは1〜5の整数のみ使用可能です。
```

---

### 出力揺れ対策のまとめ

Few-shotに加えて、出力揺れを抑えるための設定を整理します。

```
① Temperatureを下げる（0.0〜0.3を推奨）
   → AIのランダム性を下げて、安定した出力を得る
   → Difyのモデル設定から変更できる

② JSON出力を明示的に指定する
   → 「必ずJSON形式のみで回答してください」
   → 「説明文・前置きは含めないでください」

③ Few-shotで出力パターンを固定する
   → 典型例3〜5件・境界線上の例2〜3件

④ 出力バリデーションをGAS側で行う
   → JSONのパースが成功したかチェック
   → 必須キーが存在するかチェック
   → 値が想定の型・範囲内かチェック
```

④のバリデーションは「AIが間違えても気づける仕組み」です。演習のコードにも実装します。

## 4-4 Spreadsheet中心設計

### なぜSpreadsheetなのか

第3章でも触れましたが、ここでは「Spreadsheet中心設計」の具体的な作り方を説明します。

Spreadsheetを中心に置く理由を改めて整理すると：

1. **可視性**：誰でも開いて確認できる
2. **操作性**：フィルター・ソート・検索が直感的にできる
3. **変更しやすさ**：設定値の変更がセルの編集だけでできる
4. **ログとしての使いやすさ**：時系列データの管理に適している
5. **コストゼロ**：Google Workspaceに含まれている

---

### シートの設計パターン

Spreadsheetの構成は、必ず**役割別にシートを分ける**ことを推奨します。

**推奨構成：**

```
スプレッドシート全体
│
├── [入力シート]
│    目的：処理対象のデータを管理する
│    列例：ID / テキスト / 入力日時 / 処理済フラグ
│
├── [出力シート]（または入力シートに追記）
│    目的：AIの処理結果を保存する
│    列例：ID / sentiment / score / requires_action / priority / 処理日時
│
├── [ログシート]
│    目的：実行履歴とエラーを記録する
│    列例：実行日時 / 処理件数 / エラー件数 / エラー内容 / 実行時間
│
└── [設定シート]
     目的：変更頻度の高い設定値を管理する
     例：APIエンドポイント / スコア閾値 / Slack通知先 / 処理件数上限
```

---

### 設定シートの作り方

特に重要なのが**設定シートの設計**です。

こんなSpreadsheetを作ります：

| キー | 値 | 説明 |
|---|---|---|
| DIFY_API_URL | https://api.dify.ai/v1/... | DifyのAPIエンドポイント |
| SCORE_THRESHOLD_HIGH | 2 | 優先度「高」の閾値（score以下） |
| SCORE_THRESHOLD_MID | 3 | 優先度「中」の閾値 |
| SLACK_WEBHOOK_URL | https://hooks.slack.com/... | Slack通知先 |
| PROCESS_LIMIT | 50 | 1回の実行で処理する最大件数 |

GASでこの設定シートを読み込む実装：

```javascript
function getConfig() {
  const sheet = SpreadsheetApp.getActiveSpreadsheet()
    .getSheetByName('設定');
  const data = sheet.getDataRange().getValues();
  
  const config = {};
  for (let i = 1; i < data.length; i++) {
    config[data[i][0]] = data[i][1];
  }
  return config;
}

// 使い方
function main() {
  const config = getConfig();
  const apiUrl = config['DIFY_API_URL'];
  const threshold = config['SCORE_THRESHOLD_HIGH'];
  // ...
}
```

このようにすることで、**閾値やエンドポイントを変えたいときはSpreadsheetを編集するだけ**。コードに触らずに動作を変えられます。

---

### ログシートの設計

ログシートは「何かあったときに原因を調べるための証拠」です。

後から「あのとき何が起きたか」を追跡できるよう、最低限以下の情報を記録します。

```javascript
function writeLog(status, processedCount, errorCount, errorMessage) {
  const logSheet = SpreadsheetApp.getActiveSpreadsheet()
    .getSheetByName('ログ');
  
  logSheet.appendRow([
    new Date(),          // 実行日時
    status,              // "成功" or "エラー"
    processedCount,      // 処理件数
    errorCount,          // エラー件数
    errorMessage || ''   // エラー内容（あれば）
  ]);
}
```

このログがあることで：
- 「昨日の朝にエラーが出ていた」がすぐわかる
- 「先週から処理件数が減っている」に気づける
- 問題が起きたときに、担当者が状況を把握できる

## 4-5 実務AIの分業

### GAS側でやること・AI側でやること

「GASに何をやらせて、AIに何をやらせるか」の分業を正しく理解することが、安定したシステムを作るための鍵です。

**大原則：AIは「判断・生成・分析」だけに集中させる**

AIは「考えること」は得意ですが、「正確に計算すること」「ルール通りに処理すること」は苦手です。

```
GAS（人間の仕事）が担当：
✅ データの取得・整形・フィルタリング
✅ 数値の計算・集計（件数・合計・平均など）
✅ 条件分岐の実行（if/else）
✅ APIの呼び出しとレスポンスの解析
✅ Spreadsheetへの書き込み
✅ Slack・メールへの通知

AI（頭脳の仕事）が担当：
✅ テキストの分類・感情分析
✅ 要約・抽出
✅ 自然言語の生成（返信文・説明文）
✅ 曖昧な条件の判断（「このレビューは要対応か？」）
✅ 複数の要素を総合した評価（採用スクリーニングのスコアリングなど）
```

---

### よくある「間違った分業」

```
❌ 悪い例：AIに集計をやらせる
「先週の問い合わせ100件を渡すので、件数・カテゴリ別内訳・平均スコアを計算してください」

→ AIは計算が苦手。誤った数値を返す可能性がある。
→ 100件のデータを1回のAPI呼び出しで渡すと、コストが高くなる。

✅ 良い例：GASで集計し、AIに解釈させる
GASで集計：件数=100、クレーム=23%、配送問題=31%、平均スコア=3.2
AIに渡す：「この集計データの傾向を分析して、アクション提案をしてください」
→ AIは「解釈・言語化」だけに集中できる。精度が上がる。
```

```
❌ 悪い例：AIに日付計算をやらせる
「今日から7日以内に届いていない注文を教えてください」

→ AIは現在の日付を正確に知らない。計算もブレる。

✅ 良い例：GASで日付フィルタリング
GASで「今日から7日以上前の未着注文」を抽出してからAIに渡す
→ 確実に正しい対象データだけをAIが処理する。
```

---

### 分業の判断基準

どちらが担当すべきか迷ったときの判断基準：

```
GAS担当にすべき処理：
→ 正確さが100%求められる（計算・フィルタリング・日付処理）
→ ルールが明確に決まっている（○○の場合は△△する）
→ データの取得・保存・転送

AI担当にすべき処理：
→ テキストの意味や文脈の理解が必要
→ 「それっぽい」ではなく「考えた」結果が必要
→ 人間の判断を模倣したい処理
```

この分業を正しく設計することで、AIの「得意なこと」だけに集中させられるため、精度と安定性が上がります。

---

### まとめ：4章で学んだ設計原則

第4章では、実務AI設計の根幹となる考え方を5つ説明しました。

```
原則1：タスクを分割する
  → 1AIノード1タスク
  → 複数タスクを詰め込まない

原則2：JSON出力を固定する
  → スキーマを事前に設計する
  → 出力をプログラムで扱える形にする

原則3：Few-shotで揺れを抑える
  → 典型例と境界線上の例を用意する
  → 悪い例（Negative Example）も有効

原則4：Spreadsheetで可視化する
  → 設定・ログ・入力・出力を役割別シートで管理
  → 非エンジニアが運用できる設計にする

原則5：GASとAIを正しく分業する
  → AIは「判断・生成・分析」だけ担当
  → 計算・フィルタリング・データ操作はGASが担当
```

この5原則を理解した上で演習に入ると、「なぜこう設計するのか」が腑に落ちながら手を動かせます。

第5章から、いよいよ実装です。

> **第4章のまとめ**
>
> - 「入力・タスク・出力」を先に明確にすることが設計の起点
> - AIに「曖昧判断」をさせるのではなく「条件を評価させる」
> - JSON固定出力はスキーマ設計とセットで行う
> - Few-shotは典型例＋境界線上の例で精度が上がる
