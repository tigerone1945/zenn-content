## 10-1 技術より組織が難しい

### 「動くものを作った」は半分しか終わっていない

第5章から第9章まで、5つのシステムを設計してきました。

GASのコードを書いて、Difyのプロンプトを設計して、Spreadsheetに結果が書き込まれる。Slackに通知が飛ぶ。「動いた」という瞬間は、確かに達成感があります。

でも、実務の現場でAIシステムを導入するとき、「動くものを作る」は全体の半分でしかありません。

残りの半分は、**「現場で使われ続けること」** です。

そしてこの後半のほうが、技術的な実装より何倍も難しい。

---

### なぜ「組織」の問題が難しいのか

技術的な問題は、エラーメッセージが出ます。コードのどこが間違っているか、ログを見ればわかります。改善すべき方向が明確です。

組織的な問題は、エラーメッセージが出ません。

「誰も使っていない」という状態は、静かに訪れます。最初の1週間は使われていた。2週間目から件数が減ってきた。1ヶ月後には誰も触っていない——このプロセスに「エラー」は出ません。気づいたときには手遅れです。

AI導入プロジェクトの多くが失敗する理由は、技術的な問題より**組織的な問題**にあります。

演習を通じて技術を学んだ後、このチャプターで「組織に定着させるための考え方」を整理します。

## 10-2 よくある失敗

### 失敗パターン① 「作って終わる」

最も多い失敗です。

開発フェーズでは「要件定義・設計・実装・テスト」が丁寧に行われる。しかし「リリースした後どうするか」の設計が抜けていることが多い。

```
作って終わるプロジェクトの典型的な流れ：

Week1〜4：設計・実装・テスト
Week5　　：リリース・担当者への説明会
Week6　　：担当者が使い始める
Week8　　：使う人が減ってきた
Week12　 ：誰も使っていない
Week16　 ：「あのシステム、結局どうなったんだっけ」
```

「作った後」の設計がなかったプロジェクトは、リリース直後が最も使われる瞬間で、あとは右肩下がりです。

作る前に決めるべきことが3つあります。

```
① 誰がオーナーシップを持つか
   → システムに責任を持ち、改善を推進する担当者を決める
   → 「みんなで使う」は「誰も面倒を見ない」と同じ

② KPIを何にするか
   → 「週に何件処理したか」「担当者の作業時間がどれだけ減ったか」
   → 数値で効果を追えるようにする

③ 改善サイクルをいつ回すか
   → 月1回は精度確認とプロンプト調整をする
   → 改善の予定がなければ、劣化するだけ
```

---

### 失敗パターン② 現場巻き込み不足

「現場のために良いものを作ってあげた」という発想で進めたプロジェクトは、ほぼ失敗します。

```
巻き込み不足が引き起こすこと：

・「自分たちには聞かれなかった」という不満感
  → 心理的な抵抗が生まれる

・「実際の業務と合っていない」という実用性の欠如
  → 作った人の想像と現場の実態がズレている

・「使い方がわからない」という学習コストの発生
  → 説明会を1回やっただけでは定着しない

・「こんなものを入れて、仕事が増えるんじゃないか」という不安
  → AIへの誤解や脅威感が解消されていない
```

巻き込みとは「説明する」ことではありません。**「一緒に考える」** ことです。

```
✅ 良い巻き込みの例：
「このレビューの分類、毎日どのくらい時間かかってますか？」
「分類する基準って、今どんなルールで判断してますか？」
「AIが分類した結果、どんな形式だと使いやすいですか？」

→ 現場の声から設計する
→ 担当者が「自分たちのシステム」と感じる
→ 完成したとき「自分が関わったもの」として使う動機が生まれる
```

---

### 失敗パターン③ 運用設計不足

「作ったが、メンテできない」状態になるパターンです。

特に多いのが、**属人化**です。

```
属人化が起きる構造：

・AIシステムを実装したエンジニアAさんだけが内部を理解している
・Aさんが異動・退職する
・誰もメンテできなくなる
・プロンプトが古くなっても更新できない
・エラーが出ても原因がわからない
・結果として使われなくなる
```

このBook全体を通じてDifyを選んだ理由のひとつが、**「非エンジニアでもプロンプトを更新できる」** ことです。しかしツールだけでは属人化は防げません。

運用設計に必要なことは3つです。

```
① ドキュメントを残す
   → 「なぜこのプロンプトを書いたか」の意図を残す
   → 設定シートの各項目の説明を書いておく
   → 「月1回、精度確認とプロンプト更新をする」という運用手順を書く

② 担当者を複数にする
   → メインとバックアップ、最低2名が内部を理解している状態にする
   → 「Aさんしか知らない」を作らない

③ 定期的な棚卸しをする
   → 3ヶ月に1回、「まだ使われているか」「精度は維持されているか」を確認する
   → 使われていないシステムは、止めることを決断する勇気も必要
```

## 10-3 AI導入成功パターン

### 成功しているチームに共通すること

現場でAI導入がうまくいっているケースを見ると、共通するパターンがあります。

---

### 成功パターン① 小さく始めて、成功体験を積む

失敗するプロジェクトは「大きく始める」傾向があります。

```
❌ 大きく始めるパターン：
「お問い合わせ対応の全プロセスをAI化したい。
 受信から分類・担当者振り分け・返信・クローズまで全自動化。
 3ヶ月で完成させる。」

→ 要件が大きすぎて設計が長引く
→ どこかでつまずくと全体が止まる
→ 現場に届くまでに担当者のモチベーションが切れる
```

```
✅ 小さく始めるパターン：
「まず、お問い合わせのカテゴリ分類だけAIにやらせてみよう。
 2週間で動かして、現場に触ってもらう。」

→ 2週間で「動いた」という体験ができる
→ 現場から「これ、楽になった」という声が出る
→ 「次はここも自動化できないか」という要望が現場から上がってくる
→ 現場主導で改善が進む
```

最初の成功体験が、次のステップへの推進力になります。**「小さく動かして、成功体験を作る」** がAI導入の黄金律です。

---

### 成功パターン② 現場の「困り事」から始める

「AIで何かやりたい」ではなく「この業務がしんどい」から始まるプロジェクトは、定着率が高い。

```
✅ 現場の困り事から始まる例：

CSチームから：「毎週月曜の朝、先週の問い合わせを集計してレポート作るのが
               辛い。2時間かかるし、他の仕事が積み上がる」

→ 課題が明確
→ 解決されたときの効果が数値で測れる（2時間の削減）
→ 担当者が「早く使いたい」というモチベーションを持っている
```

「AIで何かやりたい」という動機でスタートしたプロジェクトは、技術的には動いても「何のために使うのか」が現場に伝わらないことが多い。

**「誰の、どんな困り事を解決するか」から逆算して設計する**。これが現場定着の出発点です。

---

### 成功パターン③ 「完璧」より「使える」を優先する

AIに100%の精度を求めると、永遠にリリースできません。

よくある議論：

```
「カテゴリ分類の精度が87%だから、まだリリースできない」

でも考えてほしいのは：
・AI導入前の人間の分類精度は何%か？
・87%の自動化で、担当者の作業時間はどれだけ削減されるか？
・残りの13%のエラーは、どう検知・修正するか？
```

87%の精度でも、人間が全件確認していた状態より業務が改善されるなら、リリースする価値があります。

「AIが間違える」ことを前提にシステムを設計するのが重要です。**間違えたとき、すぐ気づいて修正できる仕組み**（ログ・バリデーション・人間のレビュー）があれば、完璧でなくても実務で使えます。

## 10-4 DXとの関係

### AI導入を「入口」として使う

「AI導入 = DX完了」ではありません。DXの本質は業務の構造を変えることであり、AIはその手段のひとつです。

大事なのは、AI導入で生まれた変化を「ここで終わり」にしないことです。

```
効率化で止まる：
・分類作業が自動化された → 担当者の作業時間が減った

変革の入口にする：
・分類が自動化された → 担当者が本質的な顧客対応に集中できた
　　　　　　　　　　→ データが蓄積された → 傾向が見えるようになった
　　　　　　　　　　→ FAQを継続的に改善するサイクルが生まれた
```

演習を通じて各システムはSpreadsheetにデータを蓄積しています。このデータは、次の改善の入力になります。「感情スコアが新商品リリース後に急落している」「月曜の問い合わせが他の曜日の2倍ある」——AIが処理したデータが、業務改善の意思決定を支える資産に変わります。

AI導入の先に何があるかは、第11章で整理します。

## 10-5 「作れる人」から「定着させられる人」へ

本Bookを通じて、あなたはAIエージェントを「作れる人」になりました。

しかし実務の現場では、**「作れる人」より「定着させられる人」のほうが希少で、価値が高い。**

定着させるためには、実装スキルの先にある3つの領域が必要です。

```
① Guardrail（防護柵）の設計
   → AIが間違えたとき、組織を守る仕組みをどう設計するか
   → リスク管理・品質保証・倫理的なAI利用のガバナンス

② AgentOps（継続的改善）
   → 動かし続けるための監視・改善・進化のサイクル
   → AIシステムの「育て方」を知っている人が組織に必要

③ 組織設計・変革管理
   → 技術ではなく「人と組織」をどう変えるか
   → AI活用が文化として根付くための設計思想
```

これらは「技術」ではなく「戦略」の領域です。

本Bookの演習で「作る力」を身につけた今、次のステップは「組織に定着させる力」を手に入れることです。
