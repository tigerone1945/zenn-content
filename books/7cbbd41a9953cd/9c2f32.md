## 7-1 シナリオ

### 何を解決するか

ECサイトの商品登録担当者が、毎日直面している課題があります。

```
・新規登録する商品数：1日10〜50件
・1件の説明文を書く時間：15〜30分
・書く内容：短文説明 / 長文説明 / ハッシュタグ
・担当者のライティングスキルによって品質にバラつきが出る
```

説明文の品質は、購入転換率に直結します。でも、商品知識はあってもライティングが得意ではない担当者も多い。毎回「どう書けばいいか」で悩むことに時間が消えています。

このシステムは、商品情報を入力するだけで**短文説明・長文説明・ハッシュタグを自動生成**します。担当者は「ゼロから書く」ではなく「生成結果を確認・微修正する」ワークフローに移行できます。

### 完成イメージ

```
[入力]
・商品名
・商品カテゴリ
・商品の特徴（箇条書きでOK）
・ターゲット顧客
・価格帯

[出力（AI自動生成）]
・短文説明（60文字以内）：一覧ページ・広告用
・長文説明（300文字程度）：商品詳細ページ用
・ハッシュタグ（10個）：SNS投稿用
```

## 7-2 作るもの

### 機能一覧

| 機能 | 内容 |
|---|---|
| 短文説明生成 | 60文字以内のキャッチコピー的な説明文 |
| 長文説明生成 | 300文字程度の商品詳細説明文 |
| ハッシュタグ生成 | SNS投稿用のハッシュタグ10個 |
| 複数列書き込み | 生成結果を対応するSpreadsheet列に一括書き込み |

### Spreadsheetの設計

**「商品」シート（メインシート）**

| 列 | 内容 | 入力者 |
|---|---|---|
| A | 商品ID | 手動 |
| B | 商品名 | 手動 |
| C | カテゴリ | 手動 |
| D | 特徴（箇条書き） | 手動 |
| E | ターゲット顧客 | 手動 |
| F | 価格帯 | 手動 |
| G | 短文説明 | AI（自動） |
| H | 長文説明 | AI（自動） |
| I | ハッシュタグ | AI（自動） |
| J | 処理日時 | GAS（自動） |
| K | ステータス | GAS（自動） |

演習1・2と同じく、**入力列（A〜F）と出力列（G〜I）を明確に分けます**。入力列は手動入力。出力列はGASとAIが書き込む。この境界線が運用時の混乱を防ぎます。

## 7-3 Dify設計

### 「分析系」と「生成系」の違い

演習1・2は「分析系」タスクでした。与えられたテキストを読んで、分類・評価・抽出をする。答えはある程度決まっています。

演習3は「生成系」タスクです。入力情報を元に、**存在しない文章を新しく作る**。答えは1つではなく、「より良い」か「より悪いか」という評価軸になります。

この違いがプロンプト設計に影響します。

| | 分析系（演習1・2） | 生成系（演習3） |
|---|---|---|
| 出力の性質 | 決まった答えがある | 正解は複数ある |
| 精度の測り方 | 正解率で評価しやすい | 品質で評価する（主観あり） |
| プロンプトの焦点 | 判定基準の明示 | トーン・制約・ペルソナの設定 |
| Few-shotの役割 | 「このケースはこう判定する」 | 「このレベルの文章を作る」 |

生成系タスクでは、**「どんな文章を作るか」のイメージをプロンプトで明確に伝えることが最重要**です。

---

### ワークフローの全体像

```
[開始]
  入力変数：product_info / target_customer / price_range
  ↓
[LLMノード]：短文・長文・ハッシュタグを一括生成
  ↓
[終了]
  出力：JSON形式の生成結果
```

今回は1ノードで3種類の出力を生成します。短文・長文・ハッシュタグは同じ商品情報から生成するため、**文脈の一貫性を保つために1ノードにまとめる**方が品質が安定します。

ノードを分けると「短文ではAIが商品を高級感のある表現で捉えているのに、長文では機能的な説明に終始している」という一貫性の崩れが起きることがあります。

---

### プロンプト設計のポイント

**① コピーライターのペルソナを与える**

「商品説明を書いてください」より、「あなたはECサイト専門のコピーライターです」と役割を与えるほうが、生成文の質が上がります。

ペルソナを設定するときに重要なのは、**「どんな文章を書かないか」も指示すること**です。

```
良い例：
あなたはEC専門のプロコピーライターです。
・読者に「これが欲しい」と思わせる具体的な表現を使う
・「高品質」「おすすめ」「最高」などの抽象的な褒め言葉は使わない
・商品の特徴から「顧客が得られる体験」を想像して書く
```

「使わない言葉」を指定することで、生成文のクオリティが大きく安定します。

**② ターゲットをプロンプトに反映させる**

「ターゲット顧客」の情報をプロンプトに組み込みます。

```
入力されたターゲット顧客：30代・働くママ・時短を重視

↓

プロンプトに反映：
「30代の忙しいママが読んで『これ、私のためにある』と感じるような
 言葉選びをしてください。時短・便利・手間いらずといった価値観を
 自然に織り込んでください。」
```

ターゲットをただ変数として渡すのではなく、**「そのターゲットにどう届けるか」をプロンプトで解釈して指示する**ことで、生成文がターゲットに刺さるものになります。

**③ 文字数制約は「以内」より「程度」で**

「60文字以内で書いてください」と指示すると、AIは安全策として短めの文章を生成する傾向があります。「60文字程度で、最低50文字・最大70文字の範囲で書いてください」のように幅を持たせると、より情報量が豊かな生成文になります。

**④ Few-shotは「レベル感」を伝えるために使う**

生成系のFew-shotは「このような文章を生成せよ」という**レベル感・スタイルの基準**を伝えるために使います。

```
良い例（目指すレベル感）：
「片手でポンと置けば、あとはおまかせ。
 朝の忙しい時間を、少しだけ自分のために使えるようになる。」

悪い例（避けるべきレベル感）：
「高品質な商品です。使いやすく、多くのお客様から好評をいただいています。」
```

---

### 出力スキーマの設計

```json
{
  "short_description": "60文字程度の短文説明",
  "long_description": "300文字程度の長文説明",
  "hashtags": ["ハッシュタグ1", "ハッシュタグ2", "...（10個）"]
}
```

`hashtags` は配列で返させます。文字列で返すと「#商品A #商品B」のようにスペース区切りになったりカンマ区切りになったりと形式がブレるため、配列に固定してGAS側での処理を安定させます。

## 7-4 GAS実装の考え方

### 複数列入力の取り扱い

演習1・2はB列（レビューテキスト）やD列（自由記述）の1列が主な入力でした。この演習は**B〜F列の5列をまとめて入力として扱います**。

```javascript
const productInfo = [
  `商品名：${row[1]}`,
  `カテゴリ：${row[2]}`,
  `特徴：${row[3]}`,
].join('\n');
```

このように複数列の値をテキストに結合してDifyに渡します。構造化テキストにすることでLLMが各情報の意味を正確に読み取れます。

### 複数列への書き込み

生成結果のJSONをパースして、対応する列に書き込みます。

```javascript
sheet.getRange(rowIndex, 7).setValue(result.short_description); // G列
sheet.getRange(rowIndex, 8).setValue(result.long_description);  // H列
sheet.getRange(rowIndex, 9).setValue(result.hashtags.join(' ')); // I列（配列を文字列に）
```

`hashtags` はSpreadsheetに書き込む際にスペース区切りの文字列に変換します。Spreadsheetで扱いやすい形式に変換するのはGAS側の責務です。

### 生成失敗時の考え方

生成系タスクのエラーには、分析系とは異なる注意点があります。

分析系のエラーは「JSONが崩れている」「必須キーが欠けている」が主です。生成系では加えて**「生成されたが品質が低い」**というケースも発生します。

品質の低い生成文をシステムが検知するのは難しい。そのため、**生成結果は必ず担当者がレビューするプロセスを設計に組み込む**ことが重要です。GASが書き込んだ後、担当者が確認・修正してからECサイトに登録する、という2ステップのフローにします。

## 7-5 発展

### SNS投稿文の生成

ハッシュタグに加えて、Instagram・X（旧Twitter）・LINE公式アカウント向けの投稿文を生成する拡張ができます。各プラットフォームに最適な文体・文字数・絵文字の有無をプロンプトで指定します。

ただし1回のAPI呼び出しで生成する出力が増えるほど、品質が下がりやすくなります。プラットフォームごとにノードを分けるか、優先度の高いプラットフォームに絞ることを検討しましょう。

### キャンペーン文の生成

「今週の特集：夏のアウトドア特集」のようなキャンペーンテーマを変数として渡し、テーマに合わせたコピーを生成する設計にできます。キャンペーンごとに設定シートでテーマを管理する構造にすると、担当者がSpreadsheetを更新するだけでAIの生成トーンが切り替わります。

## 7-6 実務での使いどころ

### ECサイト運営

```
活用シーン：
・新商品登録時の説明文作成
・既存商品のリライト（SEO改善・リニューアル時）
・季節・キャンペーンに合わせたコピー更新

期待効果：
・説明文作成時間：1件15〜30分 → 5分以下
・ライティングスキルに依存しない品質の安定化
```

### マーケティング・SNS運用

```
活用シーン：
・SNS投稿用キャプションとハッシュタグの一括生成
・メルマガの商品紹介文の下書き生成
・広告コピーのバリエーション生成

注意点：
AIが生成した文章をそのまま使わず、担当者が確認・修正するフローを維持してください。
特に価格・在庫・スペックの記載がある場合は、必ず事実確認を行ってください。
```

---

### 設計の振り返り

```
✅ 生成系タスクの設計は「品質のイメージ」をプロンプトで伝える
   → 分析系の「正解の明示」とは異なるアプローチ

✅ コピーライターのペルソナを与える
   → 「使わない言葉」の指定が品質安定に効く

✅ ターゲット情報はプロンプトで解釈して渡す
   → ただの変数として渡すのではなく、届け方の指示に変換する

✅ 文字数制約は幅を持たせる
   → 「以内」より「程度（最低〜最大）」が情報量豊かな文章を生む

✅ ハッシュタグは配列で受け取る
   → GAS側の変換で書き込み形式を柔軟に対応できる

✅ 生成結果は必ず担当者レビューのプロセスを設ける
   → 「品質が低い」エラーはシステムで検知しにくい
```
