## 5-1 シナリオ

### 何を解決するか

あなたはECサイトの運営担当者です。

毎日届く商品レビューは、良い情報の宝庫です。品質問題の早期発見、人気商品の特定、改善のヒント——すべてがレビューに詰まっています。

しかし現実はこうです。

```
・1日に届くレビュー：50〜200件
・担当者が全件読むのにかかる時間：約2〜4時間
・実際に対応が必要なレビュー：全体の10〜20%
・残りの80〜90%：確認するだけで終わる
```

「全部読まなければいけない」という義務感が、担当者の時間を食っています。でも実際には、**「要対応のレビューだけを素早く拾い上げる」** だけで業務の大半は回ります。

このシステムを作ります。

### 完成イメージ

Spreadsheetにレビューテキストが入力されると、GASがDify APIを呼び出し、以下の情報を自動で書き込みます。

```
[入力]
・レビューテキスト（お客様が書いた文章）
・商品名

[出力（AI自動生成）]
・感情分類：ポジティブ / ネガティブ / 混合 / 中立
・星評価予測：1〜5（数値）
・問題点リスト：["梱包が雑", "色が違う"] など
・要対応フラグ：TRUE / FALSE
・優先度：高 / 中 / 低
```

「要対応フラグ = TRUE」のレビューは、Slack通知で担当者に届きます。Slackを確認するだけで、緊急対応が必要なレビューを見落とさずに済む状態を作ります。

## 5-2 作るもの

### 機能一覧

| 機能 | 内容 |
|---|---|
| 感情分析 | レビューの感情を4分類に分類 |
| 星評価予測 | 感情と内容から1〜5のスコアを予測 |
| 問題抽出 | ネガティブな言及をリスト形式で抽出 |
| 要対応判定 | 対応が必要なレビューをフラグで識別 |
| Slack通知 | 要対応レビューをリアルタイムで通知 |
| 条件色分け | 感情・優先度に応じたセルの自動着色 |

### Spreadsheetの設計

シートは役割ごとに4つに分けます。

**「レビュー」シート（メインシート）**

| 列 | 内容 | 入力者 |
|---|---|---|
| A | レビューID | 手動 |
| B | レビューテキスト | 手動 |
| C | 商品名 | 手動 |
| D | 投稿日 | 手動 |
| E | 感情分類 | AI（自動） |
| F | スコア | AI（自動） |
| G | 問題点 | AI（自動） |
| H | 要対応フラグ | AI（自動） |
| I | 優先度 | AI（自動） |
| J | 処理日時 | GAS（自動） |
| K | ステータス | GAS（自動） |

**A〜D列が「人間の仕事」、E〜K列が「AIとGASの仕事」** という明確な分担がポイントです。この境界線を意識して設計することで、「どこまで自動化されているか」が一目でわかります。

また、「ログ」シートと「設定」シートも最初から用意します。設定シートには、APIのURLやキー、閾値などをまとめて管理します。コードに直書きしないことで、設定変更がSpreadsheetだけで完結します。

## 5-3 Dify設計

### ワークフローの全体像

今回のDifyワークフローはシンプルな1ノード構成です。

```
[開始]
  入力変数：review_text / product_name
  ↓
[LLMノード]
  感情分析・スコアリング・問題抽出・要対応判定
  ↓
[終了]
  出力：JSON形式の分析結果
```

「分析・スコア・抽出・判定」を1つのLLMノードに集約しているのは、これらのタスクが **互いに文脈を参照し合う** からです。たとえば「要対応フラグ」は感情とスコアの両方を総合して判定します。独立したタスクを別ノードに分けるより、1ノードで文脈ごと処理させるほうが精度が安定します。

---

### プロンプト設計のポイント

プロンプトで押さえるべき点は4つです。

**① JSON以外を出力させない**

「必ずJSON形式のみで回答してください。説明文・前置き・コードブロック記号は一切含めないでください」と明示します。この一文がないと、AIはJSONの前後に説明文を付けてくることがあり、GAS側のパースが不安定になります。

**② 判定基準を業務ルールとして明示する**

「要対応かどうか」をAIに丸投げすると、判断がブレます。代わりに、業務ルールをそのままプロンプトに書き込みます。

```
requires_actionをtrueにする条件：
- 返品・返金・交換の要求がある
- 商品の安全性に関わる問題が言及されている
- scoreが2以下 かつ sentiment が「ネガティブ」または「混合」
```

「AIが判断する」のではなく「AIが条件を評価する」形にする。これが第4章で説明した原則の実践です。

**③ Few-shotで境界条件を教える**

典型的なポジティブ・ネガティブの例だけでなく、「デザインは良いが品質が悪い（混合で要対応）」「サイズが違うが交換不要（混合で非対応）」のような **境界線上のケース** を例示することが精度向上のカギです。「混合」というラベルだけでは、AIはrequires_actionをどちらにすべきか迷います。例を見せることで、判断の軸が固まります。

**④ 配列形式を明示する**

`positive_points` や `negative_points` は配列で返してほしいのに、AIが文字列で返すことがあります。「該当なしの場合は空配列 `[]` を使用する」と明示することで、GAS側のパース処理が安定します。

---

### 出力スキーマの設計

出力形式は事前に決め切ります。

```json
{
  "sentiment": "ポジティブ | ネガティブ | 混合 | 中立",
  "score": "1〜5の整数",
  "positive_points": ["配列"],
  "negative_points": ["配列"],
  "requires_action": "true | false",
  "priority": "高 | 中 | 低",
  "action_reason": "要対応の理由（非対応の場合は空文字）"
}
```

このスキーマを決めてからプロンプトを書き、GASのコードを書き、Spreadsheetの列を設計する。**スキーマが全体設計の起点** になります。

## 5-4 GAS実装の考え方

### 処理の流れ

GASの役割は「指揮者」です。AIに何を渡すか、結果をどこに書くか、誰に知らせるかを制御します。

```
① 設定シートから設定値を取得
② 未処理のレビューを取得（ステータスが空の行）
③ 1件ずつDify APIに送信
④ JSONレスポンスを解析・バリデーション
⑤ Spreadsheetの対象行に結果を書き込む
⑥ 要対応フラグがtrueならSlackに通知
⑦ 実行ログをログシートに記録
```

この流れ自体はシンプルです。実装で重要になるのは、③〜④の間に挟む **バリデーション** と、エラーが起きたときの **設計思想** です。

---

### API呼び出しの考え方

GASからDifyにデータを渡すときの構造はこうなります。

```javascript
const payload = {
  inputs: {
    review_text: reviewText,
    product_name: productName
  },
  response_mode: 'blocking',
  user: 'gas-review-analyzer'
};
```

`inputs` の中のキー名は、Difyの開始ノードで定義した変数名と一致させる必要があります。ここがズレると、Difyは変数を受け取れずエラーになります。

`response_mode: 'blocking'` は「結果が返るまで待つ」同期モードです。GASは非同期処理が扱いにくいため、このモードを選びます。

---

### バリデーションをなぜ入れるのか

AIの出力を「信頼する」のではなく「確認する」設計にします。

確認すべき内容は3層あります。

```
① JSONとしてパースできるか
   → AIがJSON以外のテキストを混入させることがある

② 必須キーが存在するか
   → sentimentやscoreが抜けていないか

③ 値が想定の型・範囲内か
   → scoreが1〜5の整数か、sentimentが4択のいずれかか
```

バリデーションはAIへの不信ではなく **「安全網」** です。99回は正しい出力が来ても、100回目に崩れることがある。その崩れに気づいて、適切にエラー処理できるかどうかが実務AIの品質を決めます。

---

### エラーハンドリングの設計思想

最も重要な設計原則は **「1件失敗しても全体を止めない」** ことです。

```javascript
for (const row of rows) {
  try {
    // API呼び出し・書き込み処理
  } catch (error) {
    // エラーをログに記録して、次の行へ進む
    writeErrorToSheet(row, error.message);
    continue; // ← これが重要
  }
}
```

50件のレビューを処理中に1件だけAPIエラーが出たとき、処理全体を止めてしまうと残り49件が宙に浮きます。エラーが出た行だけ「エラー」ステータスで記録して、それ以外は正常に処理を続ける設計にします。

また、エラーの内容はSpreadsheetのログシートに残します。「なぜエラーが出たか」を後から調べられる状態を作っておくことが、運用継続のための基盤です。

---

### ログ設計の考え方

実行ログに記録すべき最低限の情報は4つです。

```
・実行日時
・処理件数（何件を処理しようとしたか）
・成功件数
・エラー件数とエラー内容
```

これが残っていると、「昨日の夜中に実行したら3件エラーが出ていた」「先週から処理件数が減ってきている」といった異常にすぐ気づけます。

ログは「作った後に誰かが運用するための地図」です。最初から用意しておく習慣をつけましょう。

## 5-5 発展

### 時間トリガーで自動化する

GASには「時間トリガー」という機能があります。「毎朝9時にmain()を実行する」という設定を一度入れるだけで、以降は自動で動き続けます。

設定はGASのメニューから「トリガー」→「トリガーを追加」で行います。コードを変えずに、GUIだけで自動化が完成します。

### 条件色分けで視認性を上げる

Spreadsheetのセルに、優先度・感情に応じた背景色を設定します。

```
優先度「高」→ 薄い赤
優先度「中」→ 薄いオレンジ
ポジティブ（高評価）→ 薄い緑
```

Spreadsheetを開いた瞬間に「どのレビューに注目すべきか」がわかる状態を作ります。これだけで担当者の認知負荷が大きく下がります。

## 5-6 実務での使いどころ

### ECサイト運営

```
活用シーン：
・毎日届く大量レビューの一次トリアージ
・返金・返品リクエストを見落とさないアラート
・新商品の品質問題を早期キャッチ

期待効果：
・レビュー確認時間：2〜4時間 → 30分以下
・要対応レビューの見落とし防止
```

### カスタマーサポート（CS）

```
活用シーン：
・お問い合わせメールの感情分析と優先度振り分け
・クレーム対応の自動検知とエスカレーション

期待効果：
・対応優先度の振り分け工数を削減
・感情的なクレームの見落とし防止
```

### 品質管理

```
活用シーン：
・製品レビューから品質不具合を早期検知
・定性データ（レビュー）を定量化してトレンドを把握

期待効果：
・品質問題の発見が早まることによる被害拡大防止
```

---

### 設計の振り返り

この演習を通じて実践した設計原則を整理します。

```
✅ スキーマ設計が先
   → 出力形式を決めてから、プロンプト・コード・シートを設計する

✅ 責務の分離
   → Dify（AI判断）とGAS（業務制御）の役割を明確に分ける

✅ 業務ルールをプロンプトに明示する
   → 「要対応かどうか」の基準をAIに丸投げしない

✅ バリデーションは必ず入れる
   → AIの出力を信頼せず、確認する

✅ 1件失敗しても全体を止めない
   → エラーハンドリングは「継続」を前提に設計する

✅ ログは最初から用意する
   → 運用フェーズで必ず必要になる
```

この設計パターンは、次の演習でも繰り返し登場します。「レビュー分析」の部分を「採用スクリーニング」や「週次レポート」に置き換えるだけで、同じ構造で別のシステムが作れます。設計の型を体に入れることが、この演習の最大の目的です。
