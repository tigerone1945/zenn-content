## 6-1 シナリオ

### 何を解決するか

採用担当者の業務の中で、最も時間がかかるのが「一次スクリーニング」です。

```
・1ポジションあたりの応募数：数十〜数百件
・1件の応募書類を読む時間：5〜15分
・スクリーニングにかかる合計時間：数時間〜数日
・結果として「明らかに要件を満たしていない」応募者の割合：40〜60%
```

時間の大半が「明らかに合わない応募者を除外する」作業に消えています。本来、採用担当者が時間をかけるべきなのは「合いそうな候補者を深く見る」フェーズです。

このシステムは、一次スクリーニングの判断支援をAIに担わせることで、**採用担当者が「見るべき候補者」だけに集中できる状態**を作ります。

### 完成イメージ

応募フォーム（Googleフォーム）から届いた内容をSpreadsheetに蓄積し、GASがDify APIを呼び出して評価結果を自動で書き込みます。

```
[入力]
・応募者情報（氏名・経験年数・スキル・志望動機など）
・求人要件（職種・必須スキル・歓迎スキル・経験年数）

[出力（AI自動生成）]
・総合スコア：0〜100点
・スキルマッチ度：高 / 中 / 低
・抽出されたスキルリスト
・懸念点リスト
・推奨判定：面接推奨 / 要検討 / 見送り推奨
・判定理由
```

## 6-2 作るもの

### 機能一覧

| 機能 | 内容 |
|---|---|
| スコアリング | 応募内容と求人要件の適合度を0〜100点で評価 |
| スキル抽出 | 応募書類から該当スキルを自動でリスト化 |
| 懸念点抽出 | 経験不足・条件ミスマッチなどを抽出 |
| 推奨判定 | 面接推奨 / 要検討 / 見送り推奨の3択で判定 |
| Slack通知 | 「面接推奨」判定が出た場合に採用担当者へ通知 |

### Spreadsheetの設計

**「応募者」シート（メインシート）**

| 列 | 内容 | 入力者 |
|---|---|---|
| A | 応募ID | 自動採番 |
| B | 氏名 | フォーム自動 |
| C | 経験年数 | フォーム自動 |
| D | スキル・経験（自由記述） | フォーム自動 |
| E | 志望動機 | フォーム自動 |
| F | 総合スコア | AI（自動） |
| G | スキルマッチ度 | AI（自動） |
| H | 抽出スキル | AI（自動） |
| I | 懸念点 | AI（自動） |
| J | 推奨判定 | AI（自動） |
| K | 判定理由 | AI（自動） |
| L | 処理日時 | GAS（自動） |
| M | ステータス | GAS（自動） |

**「求人要件」シート**

ここがこの演習の設計上の重要ポイントです。求人要件を専用シートで管理します。

| 列 | 内容 |
|---|---|
| A | ポジション名 |
| B | 必須スキル（カンマ区切り） |
| C | 歓迎スキル（カンマ区切り） |
| D | 必要経験年数（最低） |
| E | 職務内容の説明 |
| F | 現在アクティブか（TRUE/FALSE） |

求人要件をSpreadsheetで管理することで、**採用ポジションが変わっても、コードを変えずに対応できます**。GASが実行時に求人要件シートを読み込んでDifyに渡す構造にします。

## 6-3 Dify設計

### ワークフローの全体像

```
[開始]
  入力変数：applicant_info / job_requirements
  ↓
[LLMノード1]：スキル抽出
  応募情報から該当スキルを配列で抽出
  ↓
[LLMノード2]：スコアリング・判定
  抽出スキル + 求人要件を照合してスコアと判定を生成
  ↓
[終了]
  出力：JSON形式の評価結果
```

今回は**2ノード構成**にします。1ノードに全部詰め込まない理由があります。

「スキル抽出」と「スコアリング・判定」は性質が異なるタスクです。スキル抽出は応募書類を読んで事実を拾う作業、スコアリングは求人要件と照合して評価する作業。この2つを分けることで、**どちらのノードで精度が落ちているかを特定しやすくなります**。

演習1の1ノード構成と比較しながら、「どこでタスクを分けるか」の判断基準を体感してください。

---

### `job_requirements` 変数の設計

この演習で最も重要な設計ポイントが、**求人要件の渡し方**です。

DifyにはGASから2つの変数を渡します。

```javascript
const payload = {
  inputs: {
    applicant_info: applicantText,    // 応募者情報（テキスト）
    job_requirements: requirementsText // 求人要件（テキスト）
  }
};
```

`job_requirements` には、求人要件シートから読み込んだ情報を**構造化されたテキスト**として渡します。

```
【ポジション】バックエンドエンジニア
【必須スキル】Python / REST API設計 / データベース設計
【歓迎スキル】AWS / Docker / チームリード経験
【必要経験年数】3年以上
【職務内容】toB向けSaaSプロダクトの機能開発および改善
```

JSONではなくテキスト形式にするのは、LLMがテキストのほうが文脈を読み取りやすいからです。逆に、**GASからDifyへ渡すときはテキスト、DifyからGASへ返すときはJSON**という使い分けを覚えておきましょう。

---

### スコア設計の考え方

スコアリングをAIに任せるとき、「0〜100点で評価してください」だけでは判断がブレます。**スコアの設計そのものをプロンプトに組み込む**ことが重要です。

```
スコア設計の例：

ベーススコア：50点

加点要素：
・必須スキルが1つ一致するごとに +10点（最大+30点）
・歓迎スキルが1つ一致するごとに +5点（最大+15点）
・経験年数が要件を満たす場合 +10点
・志望動機に具体性がある場合 +5点（主観的だがガイドラインを与える）

減点要素：
・必須スキルが1つも一致しない場合 -20点
・経験年数が要件を大きく下回る場合 -15点
・志望動機が明らかにコピペ・テンプレートの場合 -5点
```

このようにスコアの「根拠」を設計することで、AIが恣意的に点数をつけることを防ぎます。また、採用担当者が「なぜこのスコアなのか」を理解しやすくなります。

---

### プロンプト設計のポイント

**① 判定は3択に固定する**

「合否」の2択では情報が少なすぎる。「面接推奨 / 要検討 / 見送り推奨」の3択にすることで、**「要検討」という中間地帯を担当者が判断できる余地**を残します。AIが「見送り推奨」を出した案件でも、担当者が独自に判断して拾い上げることができます。

**② 懸念点は「事実の記述」に限定する**

「コミュニケーション能力が低そう」「熱意が感じられない」のような主観的な懸念点をAIに出力させると、採用バイアスにつながるリスクがあります。プロンプトで「スキルの不足・経験年数のミスマッチ・要件との不一致など、客観的な事実に基づく懸念点のみを記述してください」と制約をかけます。

**③ 判定理由は必ず出力させる**

スコアや判定だけでなく、「なぜそう判定したか」を文章で出力させます。これが採用担当者にとって最も価値ある情報です。AIの判断過程を可視化することで、担当者が「この判定は妥当か」を確認できます。

## 6-4 GAS実装の考え方

### 複数変数の送信

演習1との最大の違いは、**DifyへのAPIリクエストに複数の変数を渡す**点です。

```javascript
const payload = {
  inputs: {
    applicant_info: buildApplicantText(row),
    job_requirements: getJobRequirements()
  }
};
```

`buildApplicantText()` は応募者情報を構造化テキストに変換する関数、`getJobRequirements()` は求人要件シートからアクティブなポジションの要件を取得する関数です。

このように**「データを整形する処理」と「APIを呼び出す処理」を分ける**ことで、コードの見通しが良くなり、どちらかでバグが出たときの原因特定が容易になります。

### Slack通知の設計

通知するタイミングと内容を設計します。

```
通知条件：推奨判定が「面接推奨」の場合のみ通知

通知内容：
・応募者名・ポジション
・総合スコア
・マッチしたスキル
・判定理由
・Spreadsheetへのリンク（詳細確認用）
```

「見送り推奨」や「要検討」はSlack通知せず、Spreadsheetに記録するだけにします。**通知頻度が高すぎると、担当者が通知を無視するようになる**からです。「通知が来たら必ず確認する価値がある」という信頼感を維持することが、システムの実務定着には不可欠です。

### Script Propertiesの活用

APIキーなどの機密情報は、Spreadsheetの設定シートではなく**GASのScript Properties**に保存します。

```javascript
// Script Propertiesへの保存（一度だけ実行）
function setProperties() {
  PropertiesService.getScriptProperties().setProperties({
    'DIFY_API_KEY': 'your-api-key-here',
    'SLACK_WEBHOOK_URL': 'your-webhook-url-here'
  });
}

// 取得時
const apiKey = PropertiesService.getScriptProperties()
  .getProperty('DIFY_API_KEY');
```

Spreadsheetはチームで共有するため、APIキーを記載すると他のメンバーから見えてしまいます。機密度の高い情報はScript Propertiesへ、閾値や設定値はSpreadsheetへ、という使い分けが実務の基本です。

## 6-5 発展

### 職種別の切り替え

求人要件シートで複数のポジションを管理して、処理時にGASがアクティブなポジションを選択する設計にすると、複数ポジションの採用を並行して進めることができます。

「どのポジションに応募しているか」は、Googleフォームの選択肢として入力してもらい、その値をGASが読み取って対応する求人要件を引っ張る構造にします。

### 条件分岐による処理の最適化

全応募者に同じ処理をかけるのではなく、「経験年数が明らかに要件を下回る場合はスキル抽出をスキップしてすぐ見送り判定」のような条件分岐を入れることで、API呼び出しのコストを削減できます。

GASで事前チェックできる条件（数値の比較・必須項目の記載有無など）はGAS側でフィルタリングして、AIには「AIでしか判断できないもの」だけを渡す設計が効率的です。

## 6-6 実務での使いどころ

### 採用・一次選考

```
活用シーン：
・大量応募ポジションの一次スクリーニング自動化
・スキルマッチ度による優先度付け
・採用担当者のレビュー待ちキューの作成

期待効果：
・スクリーニング時間：数時間 → 30〜60分
・見落としリスクの低減（AIが全件チェック）
```

### HR業務全般

```
活用シーン：
・社内公募・異動申請の一次評価
・研修参加者の選定
・フリーランス・業務委託の適性評価

注意点：
AIの判定はあくまで「参考情報」です。
最終的な採用可否の判断は必ず人間が行ってください。
特に「見送り推奨」のケースでも、担当者の目で確認するプロセスを維持することを強く推奨します。
```

### 設計の振り返り

```
✅ 求人要件を外部データ（シート）で管理する
   → コードを変えずにポジション変更に対応できる

✅ 複数変数の渡し方を設計する
   → applicant_infoとjob_requirementsを別変数で渡す

✅ スコア設計をプロンプトに組み込む
   → 「なぜそのスコアか」を可視化する

✅ 懸念点は客観的事実に限定する
   → 採用バイアスのリスクをプロンプト設計で下げる

✅ 通知は「面接推奨」のみに絞る
   → 通知の信頼感を維持して、担当者が無視しない仕組みにする

✅ 機密情報はScript Propertiesへ
   → Spreadsheet共有でのAPIキー漏洩を防ぐ
```
