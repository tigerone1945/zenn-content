## 8-1 シナリオ

### 何を解決するか

多くの現場で、週次レポートはこんな形で作られています。

```
毎週月曜日の朝：
1. 先週のお問い合わせデータをCSVでダウンロード
2. Excelで件数・カテゴリ別・曜日別に集計
3. 集計結果を見ながらコメントを書く
4. PowerPointにまとめる
5. メールで配信する

→ 所要時間：1〜3時間
→ 担当者不在の場合：レポートが出ない
```

この作業の多くは「データを見てコメントを書く」という繰り返し作業です。AIが最も得意とする領域です。

このシステムを作ることで、**月曜の朝にメールを開くと、先週のサマリーレポートが届いている状態**を実現します。担当者の作業はゼロ。レポートを見て判断するだけ。

### 完成イメージ

```
毎週月曜 9:00（時間トリガーで自動起動）

↓

GASが先週のお問い合わせデータをSpreadsheetから取得・集計

↓

集計データをDifyに送信

↓

AIが傾向分析・インサイト抽出・アクション提案を生成

↓

GmailでレポートをHTML形式で送信
```

## 8-2 作るもの

### 機能一覧

| 機能 | 内容 |
|---|---|
| 自動集計 | GASが先週のデータを件数・カテゴリ・曜日別に集計 |
| 傾向分析 | AIが集計データから傾向とインサイトを文章で生成 |
| アクション提案 | AIが「次週までに取るべき行動」を提案 |
| Gmail自動送信 | 生成されたレポートをHTML形式でメール送信 |

### Spreadsheetの設計

この演習では、2つのSpreadsheetを使います。

**「お問い合わせログ」シート（入力元）**

演習1で作った感情分析システムのレビューシートと同様に、日々のお問い合わせデータが蓄積されているシートです。

| 列 | 内容 |
|---|---|
| A | 問い合わせID |
| B | 受信日時 |
| C | カテゴリ |
| D | お問い合わせ内容 |
| E | 感情スコア（演習1の結果） |
| F | 要対応フラグ |

**「週次レポート履歴」シート（出力先）**

生成されたレポートのアーカイブとして蓄積します。

| 列 | 内容 |
|---|---|
| A | 対象週（例：2024/01/08〜2024/01/14） |
| B | 総件数 |
| C | カテゴリ別件数（JSON文字列） |
| D | 要対応件数 |
| E | 生成されたレポート本文 |
| F | 実行日時 |

## 8-3 GAS実装の考え方

### この演習の最重要原則

**AIに集計をさせない。GASで集計して、AIには解釈だけさせる。**

これがこの演習の設計上の核心です。

なぜ集計をAIにさせてはいけないのか。第4章でも触れましたが、改めて具体的に考えます。

```
❌ 悪い設計：100件のお問い合わせデータを全文AIに渡して
「件数・カテゴリ別・曜日別に集計してサマリーを作ってください」

問題点：
・100件分のテキストを渡すとAPIコストが高くなる
・AIは数値の集計・計算が苦手（ハルシネーションが起きやすい）
・「50件のうちクレームが23件」という数字が正確かどうか検証できない
```

```
✅ 良い設計：GASで集計を完了させてからAIに渡す

GASが集計する：
・総件数：87件
・カテゴリ別：配送問題32件・商品不良18件・使い方質問27件・その他10件
・曜日別：月13件・火15件・水14件・木11件・金18件・土9件・日7件
・要対応件数：24件
・前週比：+12件（+16%）

この集計結果テキストをDifyに渡す
→ AIはテキストを読んで「解釈・言語化」だけに集中できる
```

GASとAIの分業を正しく理解した設計が、このシステムの品質を左右します。

---

### 日付処理の設計

「先週のデータ」を取得するために、GASで日付範囲を計算します。

```javascript
// 先週の月曜〜日曜を取得する
const today = new Date();
const dayOfWeek = today.getDay(); // 0=日曜, 1=月曜, ...
const lastMonday = new Date(today);
lastMonday.setDate(today.getDate() - dayOfWeek - 6);
lastMonday.setHours(0, 0, 0, 0);

const lastSunday = new Date(lastMonday);
lastSunday.setDate(lastMonday.getDate() + 6);
lastSunday.setHours(23, 59, 59, 999);
```

日付処理はGASが担当する典型例です。「先週の月曜から日曜まで」という条件計算をAIに任せると、実行タイミングによって結果がブレます。GASで確実に計算して、その結果をAIに渡します。

---

### 集計データのテキスト化

GASが集計した結果を、AIが読みやすい構造化テキストに変換してDifyに渡します。

```
【週次集計データ】
対象期間：2024年1月8日（月）〜 1月14日（日）

■ 全体
・総件数：87件
・前週比：+12件（+16%）
・要対応件数：24件（全体の27.6%）

■ カテゴリ別内訳
・配送問題：32件（36.8%）
・商品不良：18件（20.7%）
・使い方質問：27件（31.0%）
・その他：10件（11.5%）

■ 曜日別分布
・月曜：13件 / 火曜：15件 / 水曜：14件
・木曜：11件 / 金曜：18件 / 土曜：9件 / 日曜：7件

■ 前週との比較
・配送問題：前週22件 → 今週32件（+10件・+45%）
・商品不良：前週20件 → 今週18件（-2件・-10%）
```

このテキストをDifyに渡します。**数字はすべてGASが計算したもの**。AIはこの数字を読んで「意味」を解釈する役割だけを担います。

---

### 時間トリガーの設計

GASの時間トリガーで毎週月曜9時に自動実行します。

トリガーを設計する際に考慮すべき点が2つあります。

**実行時間の見積もり**

GASは1回の実行に6分の上限があります。データ量が多い場合、集計処理に時間がかかりすぎてタイムアウトする可能性があります。対策として、集計対象期間を限定する（直近7日分のみ処理する）か、処理を分割する設計を入れます。

**失敗時の通知**

トリガーが失敗した（エラーが出た）場合、デフォルトでは誰も気づきません。GASのトリガー設定画面で「エラー通知」をメールで受け取る設定を入れておきます。「月曜の朝にレポートが来ていない」ことに誰かが気づいて初めて障害を発見する、という状況を防ぐためです。

## 8-4 Dify設計

### ワークフローの全体像

```
[開始]
  入力変数：weekly_summary / period_label
  ↓
[LLMノード1]：傾向分析
  集計データの傾向・注目すべき変化を文章で生成
  ↓
[LLMノード2]：アクション提案
  傾向に基づいて次週の推奨アクションを生成
  ↓
[終了]
  出力：JSON形式のレポート
```

今回は2ノードに分けます。傾向分析とアクション提案は性質が異なるからです。

傾向分析は「データを読む」作業。アクション提案は「読んだ結果として何をすべきか考える」作業。前者の出力を後者のノードに引き渡す構造にすることで、アクション提案が傾向分析の内容を踏まえたものになります。

---

### プロンプト設計のポイント

**① 読者（受信者）を明確にする**

「週次レポートを作成してください」と指示するより、「このレポートを読むのは、日次の業務に追われている現場マネージャーです。3分以内に読み終えられるボリュームで、重要度の高い情報から先に書いてください」と受信者を明確にすることで、生成文の構成・ボリューム・言葉遣いが変わります。

**② 「変化」に注目させる**

前週比のデータを渡すことで、「今週は配送問題が前週比+45%と急増しています」のような変化に着目したコメントが生成されます。「絶対値」だけでなく「変化量」をデータに含めることがポイントです。

**③ アクション提案は具体的なレベルで**

「対応を検討してください」ではなく「配送業者への問い合わせ率が急増しているため、配送業者との緊急ミーティングを設定することを推奨します」くらいの具体度を出力させます。

プロンプトに「推奨アクションは、担当者が明日からすぐに動けるレベルの具体性で書いてください」と明示します。

**④ 出力構造をメール表示に最適化する**

GmailにHTMLメールとして送信するため、出力テキストはHTML形式で生成させます。`<h2>`で見出し、`<ul>`で箇条書き、`<strong>`で強調というシンプルなHTMLをFew-shotで示します。GASの `GmailApp.sendEmail()` は `htmlBody` オプションでHTML形式のメールを送れるため、AIの出力をそのままメール本文として使えます。

---

### 出力スキーマの設計

```json
{
  "headline": "今週のひとことサマリー（1文）",
  "trend_analysis": "傾向分析の本文（HTML形式）",
  "key_findings": ["注目すべきポイント1", "ポイント2", "ポイント3"],
  "recommended_actions": ["推奨アクション1", "アクション2"],
  "next_week_focus": "来週注視すべき点（1〜2文）"
}
```

`headline` をメールの件名として使い、残りをメール本文に組み立てます。GASが受け取ったJSONを元にHTMLメールを組み立てて `GmailApp.sendEmail()` で送信する構造にします。Gmailを使うことで、レポートがメールとしてアーカイブされ、後から検索・参照しやすくなるという利点もあります。

## 8-5 実務で重要な考え方

### GAS集計 × AI解釈という分業の本質

この演習は「GASがやること」と「AIがやること」の分業を最も明確に体感できる演習です。

改めて整理します。

```
GASの担当：
→ いつのデータを処理するか（日付計算）
→ 何件あるか（件数集計）
→ カテゴリごとに何件か（グルーピング）
→ 前週より増えたか減ったか（差分計算）
→ 何パーセントを占めるか（割合計算）

AIの担当：
→ この数字が意味することは何か（解釈）
→ 特に注目すべき変化はどこか（優先順位づけ）
→ 何が起きているのか（仮説の提示）
→ 次に何をすればいいか（提案）
```

数字を出すのはGAS。数字の意味を語るのはAI。この分業が成立することで、**集計の正確性とインサイトの豊かさ**が両立します。

---

### 「使われるレポート」の設計

どんなに精度が高いレポートも、読まれなければ価値ゼロです。

「使われるレポート」を設計するためのチェックリスト：

```
✅ 受け取った人が3分以内に読める量か
✅ 「で、何をすればいいの？」が書いてあるか
✅ 重要なことが先に来ているか
✅ 受け取るタイミングが適切か（月曜朝は適切、金曜夕方は微妙）
✅ 誰に届けるべきかが明確か（全員送付より、担当者だけへの通知）
```

AIが「良いレポート」を生成するためには、プロンプトでこのチェックリストの観点を伝える必要があります。AIは「使われるレポートとは何か」を知りません。設計者が教えてあげる必要があります。

## 8-6 実務での使いどころ

### 管理レポート・KPI分析

```
活用シーン：
・週次・月次のKPIレポートの自動生成
・クレーム率・解決率・対応時間などのモニタリング
・経営会議向けサマリーの下書き生成

期待効果：
・レポート作成時間：週1〜3時間 → ほぼゼロ
・担当者不在でもレポートが止まらない
```

### DX推進・業務改善

```
活用シーン：
・AIシステム自体の稼働状況レポート（「先週、何件をAI処理したか」）
・業務改善施策の効果測定レポート
・部門横断でのデータ共有

ポイント：
レポートの自動化は「DXの成果を可視化する」ツールとしても機能します。
「AI導入前後でレポート作成時間がどう変わったか」を
このシステム自体が証明できます。
```

---

### 設計の振り返り

```
✅ 集計はGAS、解釈はAIという分業を徹底する
   → AIに計算をさせない。数字の「意味」だけを語らせる

✅ 集計データは構造化テキストに変換してから渡す
   → LLMが文脈を正確に読み取れる形にする

✅ 変化量（前週比）をデータに含める
   → 「増えた・減った」に着目したコメントが生成される

✅ 受信者を明確にしてプロンプトに組み込む
   → 誰が読むかで、文体・ボリューム・構成が変わる

✅ アクション提案は「明日から動けるレベル」の具体度で
   → 抽象的な提案は読まれても行動につながらない

✅ トリガー失敗時の通知を設定する
   → 「レポートが来ていない」に気づける仕組みを作る
```
