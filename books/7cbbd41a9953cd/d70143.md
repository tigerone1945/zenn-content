## 9-1 シナリオ

### 何を解決するか

越境ECの普及により、海外からの問い合わせが増えています。

```
・問い合わせが届く言語：英語・中国語・韓国語・タイ語など複数
・現状の対応：Google翻訳でざっくり理解して、日本語で担当者に転送
・課題：翻訳の精度にムラがある / 緊急度の判断が難しい / 返信文のクオリティが低い
```

担当者が直面している問題は「翻訳」だけではありません。

- **どの言語で来たかを確認する作業**（英語なのか中国語なのか）
- **どのカテゴリの問い合わせかを判断する作業**（返品？配送？クレーム？）
- **緊急度を判断する作業**（今日中に対応が必要か、通常対応でいいか）
- **返信文を作る作業**（相手の言語で返す必要がある）

これらすべてを、一人の担当者が多言語で対応しなければならない。

このシステムは、**言語判定・分類・優先度判定・返信文生成をすべて自動化**して、担当者が「確認して送信するだけ」の状態を作ります。

### 完成イメージ

```
[入力]
・問い合わせテキスト（どの言語で来てもOK）
・顧客情報（必要最低限）

[出力（AI自動生成）]
・言語判定：ja / en / zh-CN / ko / th など
・カテゴリ：返品 / 配送 / 商品不良 / 使い方 / その他
・優先度：高 / 中 / 低
・感情スコア：1〜5
・要点（日本語）：担当者向けのサマリー
・返信文（原文と同じ言語）：そのまま送れる返信下書き
```

### 機能一覧

| 言語判定 | 問い合わせの言語を自動識別 |
| 翻訳 | 内容を日本語に翻訳して担当者向けサマリーを生成 |

**「問い合わせ」シート（メインシート）**

| 列 | 内容 | 入力者 |
|---|---|---|
| A | 問い合わせID | 自動 |
| B | 受信日時 | 自動 |
| C | 顧客名 | フォーム自動 |

## 9-3 Dify設計

### 「翻訳してから処理」ではなく「原文のまま処理」する

多言語対応システムの設計で最初に考えるのが、このアーキテクチャの選択です。

```
アプローチA：翻訳 → 処理
原文 → [翻訳ノード] → 日本語テキスト → [分析ノード] → 結果

アプローチB：原文のまま処理
原文 → [多言語対応LLMノード] → 結果（多言語で直接処理）
```

本演習では**アプローチBを採用**します。

理由は2つあります。

1つ目は**精度**。翻訳を挟むと、翻訳のミスが後続の分析に連鎖します。「返品したい」が翻訳で「返却を検討している」になると、AIが返品要求を見落とす可能性があります。現代のLLM（GPT-4o等）は多言語を直接理解できるため、翻訳を挟む必要がありません。

2つ目は**コスト**。翻訳ノードを追加するとAPI呼び出しが増え、コストと処理時間が上がります。

ただし、担当者向けの「要点サマリー」は日本語で出力させます。返信文は相手の言語で出力させる。**出力の言語を目的に応じて使い分けること**がポイントです。

---

### ワークフローの全体像

```
[開始]
  入力変数：inquiry_text / customer_name
  ↓
[LLMノード1]：言語判定・分類・優先度判定
  言語コード / カテゴリ / 優先度 / 感情スコアをJSON出力
  ↓
[LLMノード2]：要点サマリーと返信文生成
  要点（日本語）/ 返信下書き（原文と同じ言語）をJSON出力
  ↓
[終了]
  出力：2ノードの結果を統合したJSON
```

2ノードに分けている理由は、**「判断」と「生成」を分離するため**です。

ノード1は「この問い合わせは何か」を判断します。ノード2はその判断を受けて「担当者向けのサマリーと返信文を生成」します。ノード2にはノード1の出力（カテゴリ・優先度）を引き渡すことで、返信文のトーンや内容がカテゴリ・優先度に連動したものになります。

---

### プロンプト設計のポイント

**① 言語コードは標準形式で出力させる**

言語の出力を「英語」「中国語」という日本語ではなく、`en` `zh-CN` `ko` などのISOコードで出力させます。これにより、GAS側で言語ごとの条件分岐がシンプルになります。

```javascript
// 言語コードで分岐できる
if (result.language === 'en') {
  // 英語担当者のSlackチャンネルに通知
} else if (result.language === 'zh-CN') {
  // 中国語担当者のSlackチャンネルに通知
}
```

**② 返信文は「同じ言語で」を明示する**

「問い合わせと同じ言語で返信文を生成してください」と明示します。さらに「丁寧なビジネス敬語を使う」「相手の問題を共感的に受け止めてから解決策を提示する」など、**カスタマーサポートとして望ましいトーン**もプロンプトに組み込みます。

言語ごとの敬語や文化的なニュアンス（中国語では「您」を使うなど）については、LLMが適切に処理してくれます。設計者が細かく指示するより、「丁寧なビジネス文体で」と伝えるほうが自然な文章になります。

**③ 優先度の判定基準を多言語コンテキストで設計する**

「怒っている」「緊急」といった表現は言語によって異なります。英語の「I'm furious」も中国語の「非常不满」も、どちらも強い怒りを示します。プロンプトでは感情の強度を基準にするのではなく、**「何を要求しているか」という内容で優先度を判定する**ように設計します。

```
優先度「高」の条件（言語によらず内容で判断）：
・返金・返品・交換の明示的な要求がある
・安全に関わる問題の報告がある
・法的手段・消費者センターへの言及がある
・同じ顧客からの2回目以降の問い合わせである
```

### 複数Webhookの設計

Difyのアプリは1つで作りますが、GAS側の処理を言語別に振り分けます。

```
優先度「高」の問い合わせが来た場合のSlack通知先の分岐例：

英語（en）　→ #cs-english チャンネル
中国語（zh）→ #cs-chinese チャンネル
韓国語（ko）→ #cs-korean チャンネル
その他　　　→ #cs-general チャンネル（要翻訳対応）
```

GASの設定シートで言語コードとSlackチャンネルURLの対応表を管理します。

| 言語コード | Slackチャンネル名 | 担当者 |
|---|---|---|
| en | #cs-english | 田中 |
| zh-CN | #cs-chinese | 鈴木 |
| ko | #cs-korean | 外部委託 |
| default | #cs-general | チームリード |

この対応表をSpreadsheetで管理することで、担当者が変わってもコードを変えずに設定だけ更新できます。

## 9-4 GAS実装の考え方

### Spreadsheetを対応管理の中心に置く

この演習では、Slackへの通知より**Spreadsheetを対応管理のハブにする設計**を採用します。

理由は2つあります。1つ目は、多言語の問い合わせは返信下書きの確認・修正という「画面を見ながら作業する」フローが必要で、Spreadsheetのほうが作業しやすいこと。2つ目は、言語別・カテゴリ別の一覧をフィルター・ソートで素早く確認でき、対応状況の全体把握に向いていることです。

具体的な運用フローはこうなります。

```
① GASが問い合わせを処理してSpreadsheetに書き込む
② 担当者は朝・昼・夕方の3回、Spreadsheetを開いて確認する
③ 優先度「高」の行は赤背景で目立つようにしておく
④ 担当者が返信下書き（J列）を確認・修正して送信
⑤ 対応完了したらM列に担当者名を記入（対応済みの印）
```

Spreadsheetをハブにすることで、**「誰が何を対応中か」「まだ未対応の案件はどれか」が一目でわかる**対応管理表が完成します。

### 条件分岐の設計

優先度に応じた処理の分岐を設計します。

```
優先度「高」→ Spreadsheetの行を赤背景に設定 + 処理済フラグをOFF（要対応）
優先度「中」→ Spreadsheetの行をオレンジ背景に設定
優先度「低」→ 通常の背景色のままSpreadsheetに書き込み
```

この条件分岐は**GAS側で制御**します。Difyはすべての案件について同じ処理をして結果を返す。GASが結果を受け取った後、優先度を見て色分け・フラグ設定を行う。AIに「色を設定すべきか」の判断をさせるのではなく、AIの出力（優先度）を元にGASがルールベースで処理するという分業です。

### 返信下書きの扱い方

返信下書き（`reply_draft`）は、担当者がそのまま送信するのではなく**確認・修正してから送信する**フローにします。

Spreadsheetには返信下書きをJ列に書き込みますが、担当者が修正した最終版を別の列（例：N列）に記入するフローを設けます。これにより、「AIが生成した下書き」と「実際に送信した内容」の比較ができ、品質改善のフィードバックループを作れます。

## 9-5 発展

### SLA管理

SLA（Service Level Agreement：対応時間の目標）を設定して、応答遅延を検知する機能を加えます。

```
優先度「高」：4時間以内に対応
優先度「中」：翌営業日中に対応
優先度「低」：3営業日以内に対応
```

GASの時間トリガーで定期的にSpreadsheetをチェックして、SLAを超過している案件があればM列のステータスを「期限超過」に更新する仕組みを追加します。担当者がSpreadsheetを開いたとき、期限超過の行が一目でわかるようにします。

### 言語別のパフォーマンス分析

演習4（週次サマリー）と組み合わせると、「今週の英語問い合わせは17件、うち返品要求が8件（47%）で前週比+20%」のような多言語対応のKPIレポートを自動生成できます。

### 5つの演習を貫く設計パターンの総まとめ

第5章から第9章まで、5つの演習を通じて繰り返し登場した設計パターンを最後にまとめます。

```
【パターン1】スキーマ設計が起点
  出力形式を先に決め、プロンプト・コード・シートをそれに合わせて設計する

【パターン2】JSON固定出力
  AIの出力は必ずJSON形式に固定する。プログラムで扱える形にする

【パターン3】GASとAIの責務分離
  計算・フィルタリング・日付処理 → GAS
  判断・分類・生成・解釈 → AI

【パターン4】バリデーションは必ず入れる
  AIの出力を信頼せず確認する。JSONパース・必須キー・値の範囲チェック

【パターン5】1件失敗しても全体を止めない
  try-catchでエラーをキャッチして、対象行にエラー記録して次へ進む

【パターン6】Spreadsheet中心設計
  設定・ログ・入力・出力をSpreadsheetで一元管理する
  設定はSpreadsheetで変更できる設計にする
  機密情報はScript Propertiesに分離する

【パターン7】通知は絞る
  全件通知しない。「この通知は必ず見る価値がある」という信頼感を維持する

【パターン8】人間のレビューを設計に組み込む
  AIの出力をそのまま使わない。確認・承認・修正のフローを残す
```

この8つのパターンは、今後どんなAIエージェントを作るときにも繰り返し使えます。演習のテーマが変わっても、この設計パターンの型は変わりません。
