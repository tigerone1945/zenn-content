## 11-2 Python

### GASからPythonへ

GASはGoogle Workspaceに強く結びついた実行環境です。その制約の中では非常に強力ですが、次のような要件には対応しきれないことがあります。

```
GASでは難しいこと：
・大規模データ（数万件以上）の高速処理
・複雑な機械学習モデルの実行
・Google以外のサービスとの深い統合
・ローカルファイルの操作
・高度なデータ分析・可視化
```

Pythonは、これらすべてに対応できます。

しかし大事なのは、「GASからPythonに乗り換える」という考え方ではないことです。GASが得意な領域にはGASを使い続け、**GASでは実現できない領域にPythonを使う**という使い分けが現実的なアプローチです。

---

### PythonでAIエージェントを作るとは

Pythonを使うことで、GAS × Difyの構成では難しかったことができるようになります。

```
Pythonで広がる世界：

【LangChainやLlamaIndex】
・RAGの本格実装
・複数のAIモデルを組み合わせたワークフロー
・AIエージェントのより高度な制御

【FastAPI】
・AIシステムをWebAPIとして公開
・他のシステムからAPIで呼び出せる状態にする

【Streamlit / Gradio】
・AIの処理結果をWebアプリとして可視化
・非エンジニアでもブラウザで使えるインターフェースを作る

【データ処理（pandas / polars）】
・大量データの高速集計・加工
・GASでは扱いきれない規模のデータ処理
```

Pythonを学ぶ際、AIエージェント開発の文脈では**「文法を全部覚えてから始める」必要はありません**。GASで学んだJavaScriptの概念（関数・条件分岐・ループ・APIの呼び出し）は、Pythonでもほぼそのまま使えます。

「GASでできることを、Pythonでも書けるようにする」という小さな目標から始めることを勧めます。

---

### 学習の優先順位

Pythonを学ぶ順番の目安です。

```
Step1：基本文法（1〜2週間）
  → 変数・型・条件分岐・ループ・関数
  → GASと対比しながら学ぶと理解が早い

Step2：ファイル操作とAPI呼び出し（1週間）
  → CSVの読み書き / requestsでAPIを叩く
  → GASのUrlFetchApp相当のことをPythonでやる

Step3：AI連携（2〜3週間）
  → OpenAI APIをPythonから呼び出す
  → LangChainの基本
  → DifyからPythonベースの実装へ移行を体験する

Step4：Webアプリ化（2〜3週間）
  → StreamlitでAIシステムにフロントエンドをつける
  → 非エンジニアが使える状態にする
```

このBook終了後、次の3ヶ月でStep1〜3を進めるのが現実的な目標です。

## 11-3 AI分析

### データがあるなら、分析できる

演習を通じてSpreadsheetにデータが蓄積されてきました。

```
蓄積されたデータの例：
・6ヶ月分のレビュー感情スコア
・3ヶ月分のお問い合わせカテゴリ別件数
・採用スクリーニングスコアと最終採用結果
```

このデータを「ただ蓄積する」から「分析して価値を引き出す」ステップに進むのがAI分析です。

---

### ChatGPTを使ったデータ分析

最も手軽な出発点は、**ChatGPTのデータ分析機能（Code Interpreter）** を使うことです。

SpreadsheetをCSVでエクスポートして、ChatGPTに渡すだけで、グラフの生成・傾向分析・異常検知などを自然言語で指示できます。

```
ChatGPTへの指示例：
「この6ヶ月分のレビュー感情スコアのCSVを分析して、
 月別のスコア推移をグラフにしてください。
 また、スコアが急落している時期があれば特定してください。」
```

Pythonやデータ分析の専門知識がなくても、**AIを使ってデータを分析できる**時代です。まずここから始めることを勧めます。

---

### BI・可視化ツールへの接続

データ分析を継続的なモニタリングに発展させるなら、BI（Business Intelligence）ツールの活用が有効です。

```
Googleが提供する無料のBI：
・Looker Studio（旧Google Data Studio）
  → SpreadsheetのデータをリアルタイムでダッシュボードUI化
  → 誰でもブラウザで見られる
  → コーディング不要

活用イメージ：
・週次のお問い合わせ件数トレンドグラフ
・感情スコアの分布ヒストグラム
・要対応案件の未処理件数モニタリング
```

演習のSpreadsheetとLooker Studioを繋ぐだけで、リアルタイムのKPIダッシュボードが完成します。コーディングは不要です。

---

### マーケティング分析との接続

レビュー・問い合わせデータの分析は、マーケティング施策と接続できます。

```
分析から施策への流れの例：

分析：「Aという商品のレビューで『梱包』に関するネガティブ評価が急増している」
施策：「梱包の見直し → レビュー品質の改善 → 転換率の向上」

分析：「英語問い合わせのうち40%が『返品ポリシーの確認』」
施策：「英語FAQページに返品ポリシーを目立つ場所に掲載 → 同問い合わせが30%減」
```

AIが自動分類・自動分析した結果が、**マーケティングPDCAの入力データになる**というサイクルが生まれます。

## 11-4 AWS

### なぜAWSが必要になるのか

GASには実行時間の上限（6分）があります。Difyのクラウド版には処理件数の上限があります。Spreadsheetは大規模データ処理に向いていません。

本格的な運用規模になると、これらの制約が壁になります。その壁を超えるための選択肢がAWSです。

```
AWSで解決できること：
・GASの6分制限 → Lambda（サーバーレス関数）で時間制限を緩和
・Difyのクラウド制限 → Difyをセルフホストして制限を取り除く
・Spreadsheetの容量制限 → RDS（データベース）でデータを管理
・常時起動が必要な処理 → EC2（仮想サーバー）で24時間稼働
```

ただし、**最初からAWSは必要ありません**。

GAS × Difyで問題が出てきてから、AWSを検討すれば十分です。「まずGAS × Difyで動かして、スケールが必要になったときにAWSへ移行する」という段階的なアプローチが現実的です。

---

### 学ぶ順番

AWSを学ぶ際の優先順位の目安です。

```
優先度高（AIシステム運用で最初に使う）：
・Lambda：イベント駆動のサーバーレス関数。GASの代替として使える
・S3：ファイルストレージ。ログ・CSVの保存先として使える
・API Gateway：LambdaをAPIとして公開するための入口

優先度中（規模が大きくなったら）：
・RDS：RDBMSとしてのデータ管理。Spreadsheetの代替
・CloudWatch：ログ監視とアラート設定

優先度低（当面は不要）：
・SageMaker：機械学習モデルの学習・デプロイ
・Bedrock：AWSのAIサービス群
```

このBookの読者に最初に必要なのは、Lambdaの基本です。「GASで書いていた処理をLambdaで書く」という置き換え実験から始めることを勧めます。

## 11-5 今後のロードマップ

### 3つのキャリアパスと対応する学習

GAS × Difyを出発点として、どの方向に進むかによって、学ぶべきことが変わります。

---

**パス①：業務改善・DX推進のスペシャリスト**

組織内でAI活用を推進する役割を担うパスです。

```
向いている人：
・エンジニアよりもビジネス側での貢献に興味がある
・社内のAI活用をリードしていきたい

おすすめの学習ロードマップ：
① AI分析（Looker Studio / ChatGPT分析）
② Python基礎（Pandas・データ処理）
③ クラウド概念（AWSの基礎）
④ プロジェクトマネジメント・チェンジマネジメント

目標：
「AI活用の設計と推進ができる人材」
```

---

**パス②：AIアプリ開発者**

AIを使ったプロダクト・ツールを自分で作れるエンジニアになるパスです。

```
向いている人：
・コードを書くことが好き・得意
・自分でWebアプリやAPIを作りたい

おすすめの学習ロードマップ：
① Python本格習得（基本文法 → API連携 → LangChain）
② Webアプリ化（FastAPI / Streamlit）
③ AWS（Lambda / S3 / API Gateway）
④ LLMの詳細（プロンプトエンジニアリング / RAG本格実装）

目標：
「AIシステムをゼロから設計・実装・デプロイできる人材」
```

---

**パス③：AI×ドメインのハイブリッド人材**

特定の業界・業務領域の深い知識とAI実装力を組み合わせるパスです。

```
向いている人：
・EC・HR・マーケ・カスタマーサポートなど特定領域の業務経験がある
・その業務をAIで変えることに興味がある

おすすめの学習ロードマップ：
① 自分の業務領域でGAS × Difyの演習を応用する
② ドメイン特化のデータ分析を学ぶ
③ Pythonで実装の幅を広げる
④ 自分の業務改善事例をアウトプット（Zenn / LinkedIn / 社内発表）

目標：
「○○業界のAI活用に詳しい実装者」という差別化ポジション
```

---

### アウトプットし続けることの重要性

技術は「学んだだけ」では身につきません。**作って・発表して・フィードバックをもらう**サイクルを回すことで定着します。

```
おすすめのアウトプット先：

Zenn：
→ 実装したシステムの設計を記事にする
→ 「GAS × DifyでXXを作った」という実践記録は読まれやすい

GitHub：
→ GASのコードを公開する
→ ポートフォリオとして機能する

社内発表：
→ 「AI導入でXX業務の工数がYY時間削減された」という数値を語る
→ 実績があると信頼が生まれ、次の案件につながる
```

このBookを読み終えた段階で、すでに「語れる経験」があります。第5〜9章で設計した演習の設計思想と実装アプローチを、自分の言葉でZennやSNSに書いてみてください。

書くことで理解が深まります。そして書いた記事が、次の学習機会や仕事の機会を引き寄せます。
